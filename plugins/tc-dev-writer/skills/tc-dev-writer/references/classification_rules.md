# 분류 자동 판단 룰

`선택` (버그/개선)과 `우선순위` (당장/상/중/하)를 자동으로 결정하는 기준. 사용자에게 묻지 않고 자체 판단해야 한다.

## §1. 선택 자동 판단: 버그 vs 개선

### 1차 판단 기준

> 정상 동작에서 벗어났는가?

- **벗어났다** → **버그**
- **벗어나지 않았지만 더 나아질 수 있다** → **개선**

### 버그 신호 (강한 → 약한 순)

| 신호 | 예시 |
|------|------|
| 크래시/튕김/멈춤 | "튕김", "강제 종료", "프리즈" |
| 진행 불가 | "안 됨", "버튼이 안 눌림", "화면이 안 나옴" |
| 잘못된 값/표시 | "보상이 다르게 나옴", "수치가 이상함", "표시가 깨짐" |
| 오류 메시지 | "에러 토스트", "오류 코드 표시" |
| 누락 | "안 들어옴", "보상이 안 옴", "UI가 사라짐" |
| 잘못된 상태 | "꼬임", "이상한 상태로 남음", "초기화 안 됨" |
| 의도와 다른 동작 | "원래는 X여야 하는데 Y가 됨" |

### 개선 신호

| 신호 | 예시 |
|------|------|
| 추가 요청 | "버튼 추가", "기능 추가", "표시 추가" |
| 편의성 | "더 편하게", "쉽게", "한 번에" |
| 일관성 | "다른 화면처럼", "통일", "맞춰주세요" |
| 직관성 | "이해하기 어려움", "혼란스러움", "직관적이지 않음" |
| 성능 | "느림", "로딩 길어짐", "더 빠르게" |
| 시각적 폴리시 | "예쁘게", "디자인 변경", "톤 조정" |

### 혼합 신호 처리

신호가 섞이면(예: "버튼이 잘 안 보여서 못 찾겠어요") 1차 기준(정상 동작에서 벗어났는가)으로 돌아간다.

- "버튼이 안 보임" 자체가 정상이 아니면 → 버그
- 버튼은 정확히 표시되지만 다른 요소에 묻혀 인지되지 않음 → 개선 (UI/UX 인지성 개선)

명백히 모호하면 사용자에게 짧게 묻는다 (`references/clarification_questions.md` 참조).

### 자동 판단 결과 표기

판단 결과를 사용자에게 보고할 때 한 줄로 근거를 적는다:

- "버그 (정상 동작에서 벗어남: 진행 불가)"
- "개선 (동작은 정상, UX 일관성 개선 요청)"
- "버그 (실제 동작이 기대 동작과 다름)"

## §2. 우선순위 자동 판단

사용자 인자에 `당장`, `상`, `중`, `하` 중 하나가 있으면 그대로 사용한다. 없을 때만 아래 룰을 적용한다.

### 우선순위 단계 정의

| 옵션 | 의미 | 트리거 |
|------|------|--------|
| **당장** | Blocker — 즉시 처리. 진행 불가, 금전/데이터 손실, 다수 유저 영향 | 결제 실패, 크래시 다발, 핵심 플로우 차단, 데이터 손실 |
| **상** | Critical — 빠르게 처리. 정상 사용 어려움 | 핵심 기능 일부 차단, 우회 불가, 사용자 인지 가능한 결함 |
| **중** | Major — 다음 스프린트. 사용 가능하나 일부 비정상 | 우회 가능한 버그, 사용성 떨어지는 개선, 일부 케이스 한정 |
| **하** | Minor — 시간 날 때. 사소한 결함/취향성 | UI 미세 어긋남, 엣지 케이스, 사소한 폴리시 |

### 자동 판단 의사결정 트리

```
1. 진행 불가 / 크래시 / 결제 손실 / 데이터 손실?
   YES → 당장
   NO  → 2번
2. 정상 사용이 사실상 어려운가 (우회 수단 없음, 핵심 기능 일부 차단)?
   YES → 상
   NO  → 3번
3. 다수 사용자에게 영향이 있거나, 우회 가능하지만 분명한 결함인가?
   YES → 중
   NO  → 4번
4. 사소한 UI 어긋남 / 엣지 케이스 / 취향성 / 단발성 관찰?
   YES → 하
```

### 버그 vs 개선에 따른 디폴트

- **개선** TC는 기본적으로 `중` 이하: 사용자가 명시 안 했고 영향도가 분명히 크지 않으면 `중` (편의/일관성) 또는 `하` (취향/사소)
- **버그** TC는 영향도에 따라 모든 범위 가능. 단, "당장"은 "지금 안 하면 큰일 남" 수준에만 부여 (의사결정 트리 1단계).

### 신호 키워드 (참고)

| 우선순위 | 트리거 키워드 |
|----------|---------------|
| 당장 | "튕김", "결제 안 됨", "진행 불가", "전부 안 됨", "다수 유저", "출시 막힘" |
| 상 | "핵심 기능", "우회 불가", "매번 발생", "이전 단계로 못 감" |
| 중 | "가끔", "특정 조건", "우회 가능", "편의성" |
| 하 | "사소한", "잘 안 보임", "취향", "엣지 케이스", "1회만 관찰" |

키워드가 없으면 의사결정 트리만 적용한다. 키워드는 보조 지표.

### 자동 판단 결과 표기

판단 결과를 사용자에게 보고할 때 근거를 한 줄로:

- "당장 (크래시 — 진행 불가)"
- "상 (핵심 플로우 차단, 우회 수단 없음)"
- "중 (가끔 발생, 우회 가능)"
- "하 (사소한 UI 어긋남)"

## §3. 분류 시 절대 하지 말 것

- 선택(버그/개선)을 사용자에게 묻기 — 자동 판단해야 한다. 단, 정말 양쪽 어느 쪽인지 본질적으로 모호한 경우만 짧게 확인.
- 우선순위가 사용자 입력으로 주어졌는데 "내가 보기엔 이게 더 맞다"고 임의로 덮어쓰기 — 입력값을 그대로 사용하고, 의문이 있으면 결과 보고에 한 줄 첨언 ("입력하신 '하'로 등록했지만 영향도가 커 보입니다. 필요 시 노션에서 직접 조정해 주세요.")
- "당장"을 가볍게 부여하기 — 정말 즉시 처리해야 하는 상황에만. 남발하면 진짜 당장 처리할 항목이 묻힌다.
- 키워드 매칭만으로 판단하기 — 의사결정 트리(영향도/우회 가능성)가 본질이고 키워드는 보조.
