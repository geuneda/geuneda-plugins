# 부족한 정보 역질문 패턴 (AskUserQuestion 전용)

TC 내용이 개발자가 착수할 수 있는 형태가 아닐 때 사용자에게 보충 질문을 던지는 패턴 모음. **반드시 `AskUserQuestion` 도구로만** 질문하며, 텍스트로 1, 2, 3 나열하지 않는다.

## 역질문의 첫 번째 원칙

> 의심되면 무조건 묻는다. "조금이라도 애매하면" 묻는 쪽이 기본값.

`(추정)` 메우기는 폐기. Step 2의 4개 항목(항목/재현조건/재현절차/기대vs실제) 중 하나라도 사용자 메시지로부터 단정 못하면 즉시 `AskUserQuestion` 호출.

## `AskUserQuestion` 도구 제약

- 1회 호출에 **1~4개 질문**.
- 각 질문에 **2~4개 옵션** (그 외에 "Other"이 자동 추가됨 — 사용자가 직접 입력 가능).
- `multiSelect: true`로 다중 선택 허용 가능 (디바이스/OS 등).
- `header`는 매우 짧은 라벨 (≤12자, 예: "화면", "OS", "빈도").

부족한 카테고리가 5개 이상이면 우선순위 높은 4개만 먼저 묻고, 답변 후 2차로 더 묻는다. **3차까지는 허용**, 그 이상 왕복은 피한다.

## 카테고리별 질문 템플릿

각 카테고리는 [tc_writing_guide.md](tc_writing_guide.md)의 4가지 항목에 대응한다. 모든 옵션은 도메인에서 흔히 나오는 답을 채운다 — 회피성("잘 모르겠음")은 절대 옵션에 넣지 않는다.

### A. 항목/대상

```json
{
  "question": "어떤 화면 또는 기능에서 발생/적용되나요?",
  "header": "화면",
  "options": [
    { "label": "타이틀/로비 화면", "description": "앱 진입 후 가장 처음 보는 화면 또는 게임 시작 전 메인 로비" },
    { "label": "스테이지/전투 진행 중", "description": "실제 게임플레이가 일어나는 화면" },
    { "label": "상점/인벤토리", "description": "구매·아이템 관리 관련 화면" },
    { "label": "설정/계정", "description": "환경설정, 로그인, 계정 연동 등" }
  ],
  "multiSelect": false
}
```

도메인이 더 좁다면 보기를 그 도메인에 맞춰 바꾸기. 예: 보스전 관련이면 "1챕터 보스", "5챕터 보스" 등.

### B. 디바이스/OS

```json
{
  "question": "어떤 디바이스/OS에서 확인되었나요?",
  "header": "OS",
  "options": [
    { "label": "iOS", "description": "iPhone/iPad" },
    { "label": "Android", "description": "삼성·구글·기타 안드로이드 기기" },
    { "label": "양쪽 모두", "description": "iOS와 Android 모두에서 재현" },
    { "label": "에디터/PC", "description": "Unity 에디터 또는 PC 빌드에서 발생" }
  ],
  "multiSelect": true
}
```

### C. 재현 절차/빈도

```json
{
  "question": "얼마나 자주 발생하나요?",
  "header": "빈도",
  "options": [
    { "label": "매번 (100%)", "description": "동일 조건에서 항상 재현" },
    { "label": "가끔 (절반 이하)", "description": "동일 조건이어도 재현이 들쭉날쭉" },
    { "label": "1~2번만 관찰", "description": "특정 상황에서 한두 번 본 케이스" }
  ],
  "multiSelect": false
}
```

재현 절차가 한 줄로 환원 안 될 때는 위 도구로 묻기 어렵다 (옵션화가 어려움). 이 경우는 "재현 절차를 한 단계씩 풀어주실 수 있을까요?" 식의 옵션 없는 질문을 만들고 옵션에 흔한 진입 경로 2~3개를 두어 사용자가 Other로 자세히 답하게 유도한다.

### D. 기대 동작

```json
{
  "question": "원래는 어떻게 동작해야 한다고 생각하셨나요?",
  "header": "기대",
  "options": [
    { "label": "정상 진입/전환", "description": "다음 화면이나 다음 액션으로 자연스럽게 넘어감" },
    { "label": "구체적 결과 표시", "description": "특정 값/팝업/결과가 표시되어야 함" },
    { "label": "오류 메시지 안내", "description": "실패해도 사용자에게 사유가 표시되어야 함" }
  ],
  "multiSelect": false
}
```

도메인이 좁으면 보기를 도메인 결과에 맞춰 교체.

### E. 버그/개선 판단이 모호할 때

```json
{
  "question": "이 이슈는 동작 자체 결함(버그)인가요, 개선 제안(개선)인가요?",
  "header": "유형",
  "options": [
    { "label": "버그", "description": "원래 동작이 정해져 있는데 다른 결과가 나옴" },
    { "label": "개선", "description": "현재 동작은 정상이지만 더 편하게/일관성 있게 바뀌면 좋겠음" }
  ],
  "multiSelect": false
}
```

이건 자동 판단이 명백히 불가능할 때만 (자동 판단 우선).

### F. 영향 범위 (우선순위 미입력 시만)

```json
{
  "question": "이 이슈가 사용자 진행에 어느 정도 영향을 주나요?",
  "header": "영향",
  "options": [
    { "label": "진행 불가", "description": "크래시/결제 손실 등으로 더 못 함" },
    { "label": "정상 사용 불가", "description": "핵심 기능을 못 쓰지만 다른 길은 있음" },
    { "label": "일부 기능 비정상", "description": "우회 가능한 불편" },
    { "label": "사소한 UI/엣지", "description": "발견하기 어려운 케이스" }
  ],
  "multiSelect": false
}
```

우선순위가 입력으로 들어왔다면 묻지 않는다.

## 묶어 묻기 예시

입력: `"게임 시작 누르면 튕김"`, 우선순위 `상`.

이 경우 항목/디바이스/재현절차 셋 다 불명 → 한 번의 `AskUserQuestion` 호출에 4개 질문까지 묶음:

```jsonc
[
  { /* A. 항목 */ },
  { /* B. OS, multiSelect: true */ },
  { /* C. 빈도 */ },
  { /* 재현 절차 - 진입 경로 옵션화 */ }
]
```

이렇게 1회 호출로 사용자 답변 받기.

## 질문 톤

- **비난 금지**: "정보가 부족합니다" 대신 "확인이 필요합니다".
- **개발 용어 최소화**: "엣지 케이스" "스택트레이스" 같은 용어 빼기.
- **option은 답하기 쉬운 보기**로: 사용자가 클릭만으로 답할 수 있게.
- **사용자 원문 인용**: 사용자가 쓴 표현을 따옴표로 인용해 어떤 부분을 묻는지 명확히.

## 절대 하지 말 것

- 텍스트 메시지로 "1. ... 2. ..." 식 역질문 (도구 사용 의무)
- 옵션에 "잘 모르겠음" / "상관없음" 같은 회피지를 넣기 (회피는 Other 자동 추가가 담당)
- `AskUserQuestion` 응답 받기 전에 노션 도구 호출
- 4개 초과 질문을 한 호출에 끼워넣기 (도구 제약 위반)

## 한계 — 3회 이상 왕복 시

같은 사용자와 3번 이상 `AskUserQuestion`을 반복했는데도 여전히 불명한 부분이 있다면, 사용자에게 양해를 구하고 받은 정보로 초안을 작성해 Step 8 확인 단계에서 직접 수정하도록 위임한다. 무한 ping-pong은 피한다.
