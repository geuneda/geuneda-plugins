# 부족한 정보 역질문 패턴

TC 내용이 개발자가 착수할 수 있는 형태가 아닐 때 사용자에게 보충 질문을 던지는 패턴 모음. 한 번의 왕복으로 끝낼 수 있도록 질문을 잘 묶어야 한다.

## 역질문의 첫 번째 원칙

> 사용자 왕복은 가능한 한 1회로 끝낸다.

부족한 항목이 여러 개면 한 번에 묶어 묻는다. 1번 질문 답변 후 또 묻고, 또 답변받고 또 묻는 식의 ping-pong은 사용자 인내를 갉아먹는다. 부족한 항목을 처음에 모두 식별해서 묶어 던지자.

## 질문 포맷

```
요청하신 TC를 등록하기 전에 몇 가지 확인이 필요합니다:

1. {질문}
   {필요 시 보기/예시}
2. {질문}
   {필요 시 보기/예시}
3. ...

답변 주시면 그 내용을 반영해 TC를 구체화해 등록하겠습니다.
```

## 카테고리별 질문 템플릿

각 카테고리는 [tc_writing_guide.md](tc_writing_guide.md)의 4가지 항목(항목/재현 조건/재현 절차/기대 vs 실제)에 대응한다.

### A. 항목/대상이 모호함

> "어떤 화면 또는 기능에서 발생/적용되는 건가요?"

- 보기 예시 제공:
  ```
  - 상점 화면? 인벤토리 화면? 메인 화면?
  - 특정 캐릭터? 모든 캐릭터?
  - 특정 모드(스토리/PvP/이벤트)?
  ```

### B. 재현 조건 부족

> "어떤 환경에서 발생하나요?"

세부 질문 (해당하는 것만 골라서):
- 디바이스/OS: "iOS인가요, Android인가요? 가능하면 OS 버전도 알려주세요."
- 계정: "신규 계정인가요, 기존 계정인가요? 특정 진행도가 필요한가요?"
- 진입 경로: "어떤 화면에서 어떻게 들어갔을 때 발생했나요?"
- 네트워크: "와이파이/LTE 등 네트워크 상태와 관련 있을까요?"
- 시점: "특정 이벤트 기간이거나 점검 직후 같은 타이밍 이슈인가요?"

### C. 재현 절차 부족 또는 빈도 모호

> "어떤 순서로 했을 때 발생하나요?"

- 단계로 풀어달라 요청:
  ```
  1. {시작 화면}에서
  2. {버튼/액션}
  3. {다음 액션}
  ... 형태로 풀어주실 수 있을까요?
  ```
- 빈도 확인: "매번 발생하나요, 가끔 발생하나요? 가끔이라면 몇 번 시도 중 몇 번 정도인지 감이 있으신가요?"

### D. 기대 동작 명시 안 됨

> "원래는 어떻게 동작해야 한다고 생각하셨나요?"

- 보기 제공:
  ```
  예를 들어 "구매 모달이 뜨고 결제 진행" 같은 식으로요.
  ```
- 개선 TC라면: "어떻게 바뀌면 좋겠는지 구체적으로 알려주시면 도움이 됩니다."

### E. 버그 vs 개선 모호

> "이게 동작 자체의 문제(버그)인가요, 아니면 동작은 정상이지만 더 개선되면 좋겠다는 제안(개선)인가요?"

- 보기:
  ```
  - 버그: "원래는 X여야 하는데 Y가 됨"
  - 개선: "현재 X로 동작하지만 Y처럼 바뀌면 좋겠음"
  ```

### F. 영향 범위 모호 (우선순위 자동 판단을 위함)

우선순위가 입력되지 않았고 영향 범위가 불분명할 때만 추가:

> "이 이슈가 사용자 진행에 얼마나 영향이 있을까요? 진행이 막힐 정도인지, 우회 가능한 불편 정도인지?"

이 질문은 우선순위 트리거 키워드가 전혀 없을 때만. 보통은 절차/기대 동작이 명확해지면 자동 판단 트리로 결정 가능하다.

## 모아 묻기 예시

### Bad (3번 왕복)

```
[1차] "어떤 화면인가요?"
사용자: "보스전이요"
[2차] "어떤 보스인가요?"
사용자: "5챕터 보스"
[3차] "OS는요?"
```

### Good (1회로 끝)

```
요청하신 TC를 등록하기 전에 몇 가지 확인이 필요합니다:

1. 어떤 보스전인가요? (챕터 번호 또는 보스 이름)
2. 어떤 디바이스/OS에서 발생했나요? (예: iOS 17, Android 14 등)
3. 정확한 재현 절차를 알려주실 수 있을까요?
   예: "보스전 진입 → 일반 공격으로 체력 50% 이하 → 궁극기 버튼 탭 → 아무 반응 없음"
4. 매번 발생했나요, 아니면 한두 번만 관찰됐나요?

답변 주시면 그 내용을 반영해 TC를 구체화해 등록하겠습니다.
```

## 질문 톤

- **비난 금지**: "정보가 부족합니다" 같은 비난조 대신 "확인이 필요합니다" 같은 협업 톤
- **개발 용어 최소화**: "스택트레이스", "엣지 케이스" 같은 용어는 빼고 "어떤 상황에서 발생", "특정 조건" 같은 평이한 표현
- **예시 동봉**: 추상적 질문보다 구체적 보기 예시를 함께 제공해서 사용자가 답하기 쉽게
- **사용자 원문 인용**: "말씀하신 '튕김'이 강제 종료를 의미하나요, 아니면 화면이 멈춤을 의미하나요?" 식으로 원문을 따옴표로 인용

## 역질문하지 않아도 되는 경우

다음은 추정으로 채워도 무방하다 (`(추정)` 표시):

- 디바이스가 명시 안 됐지만 OS 일반으로 발생 가능한 일반 UI 버그 → "[재현 조건] 양 OS (추정)"
- 빈도가 명시 안 됐지만 "안 됨" 같은 구조적 결함 → "[빈도] 매번 (추정)"
- 절차가 1단계로 충분한 경우 ("로그인 화면 진입 시 뒤로가기 버튼 안 보임") → 보이는 그대로 적기

## 한계 - 3회 이상 왕복 발생 시

같은 사용자와 3번 이상 묻고 답하기를 반복하면:

```
이 정도까지 받은 정보로 일단 TC를 작성해 보여드릴게요.
부족하거나 잘못된 부분이 있으면 알려주시면 수정해서 등록하겠습니다.
```

식으로 일단 정리한 초안을 보여주고 사용자가 확인하는 방향으로 전환한다. 무한 ping-pong을 피한다.
