# TC 등록 스킬 (tc-dev-writer)

사용자가 던진 짧은 TC 한 줄을 받아 개발자가 그대로 착수할 수 있는 형태로 구체화한 뒤 노션 TC 데이터소스에 한 행을 추가한다.

## 핵심 원칙

- **추측 금지, 의심되면 무조건 역질문**: 4가지 핵심 항목(항목/재현조건/재현절차/기대vs실제) 중 어느 하나라도 사용자 메시지에서 직접 인용할 수 없거나 단정적으로 결론낼 수 없는 부분이 있으면 **반드시** 사용자에게 묻는다. "조금이라도" 애매하면 묻는 쪽이 기본값이다. `(추정)` 표시로 메우는 옵션은 폐기.
- **역질문은 반드시 `AskUserQuestion` 도구로**: 텍스트로 1, 2, 3 나열하지 않고 객관식 옵션이 붙은 `AskUserQuestion`을 사용한다. 부족한 항목이 여러 개면 1회 호출에 최대 4개까지 묶어 던진다 (도구 제약). 옵션은 답하기 쉬운 2~4개를 제시, "Other"은 자동 추가됨.
- **역질문 전엔 노션 미수정**: `AskUserQuestion` 응답을 받기 전에는 어떤 노션 도구도 호출하지 않는다.
- **선택(버그/개선)은 자동 판단**: 사용자에게 묻지 않는다. `references/classification_rules.md`의 룰을 따른다.
- **우선순위는 입력 우선**: 사용자가 인자로 넘겼다면 그대로 사용한다. 없을 때만 자동 판단한다.
- **번호는 데이터소스의 다음 정수**: 기존 페이지의 `번호` 중 정수 최대값 + 1. 처음 등록이면 1. 명명 규칙(`TC-...-슬러그`)은 사용하지 않는다.
- **다른 데이터소스 미수정**: 기획(`collection://3662dc3e-f514-80eb-afe5-000b7e302c1d`) / QA(`collection://3662dc3e-f514-801c-9d9b-000ba701f95b`) 등 TC 외 데이터소스는 절대 읽거나 쓰지 않는다.
- **노션 페이지 본문에 코드 작성 금지**: TC는 명세지 구현이 아니다.

## 입력 인자

스킬은 두 개의 인자를 받는다:

| 위치 | 이름 | 필수 | 설명 |
|------|------|------|------|
| 1 | TC 내용 | O | 자연어 한 줄~여러 줄. 사용자가 발견한 현상/제안. |
| 2 | 우선순위 | X | `상`, `중`, `하`, `당장` 중 하나. 생략 시 자동 판단. |

인자가 비어 있으면 사용자에게 다시 묻는다. 우선순위 인자가 `상/중/하/당장` 외 값이면 한 번 확인한 뒤 자동 판단으로 폴백한다.

## 워크플로우

### Step 1: 프로젝트 설정 확인

`references/project_config.md`에 따라 데이터소스 URL과 컬럼 매핑을 확보한다.

1. 현재 작업 디렉토리에서 `./.tc-dev-writer.json` 확인
2. 없으면 `~/.claude/tc-dev-writer.config.json` 확인
3. 둘 다 없으면 사용자에게 노션 TC 데이터소스 URL을 묻고 새로 작성

설정이 없는 첫 실행에서만 묻는다. 그 다음부터는 자동으로 로드한다.

### Step 2: TC 내용 1차 분석

받은 TC 내용을 다음 4가지 항목에 대해 채워본다 (`references/tc_writing_guide.md` 참조):

1. **항목/대상**: 어떤 기능/시스템/화면에서 발생/적용되는지
2. **재현 조건**: 어떤 상태에서 발생하는지 (디바이스, 계정, 진입 경로)
3. **재현 절차**: 1, 2, 3 식 단계 (`개선`이면 "현재 동작" 단계)
4. **기대 동작 vs 실제 동작**: 무엇이 어떻게 다른지 (`개선`이면 "개선 후 동작")

각 항목에 대해 다음 3개 라벨 중 하나로 분류한다:

- **확정**: 사용자 메시지에서 직접 인용 가능. 그대로 사용.
- **불명**: 사용자 메시지로부터 단정할 수 없다 (추측 필요). → **즉시 Step 3 역질문 후보**
- **자명**: 추측이 아니라 도메인 상식으로 단 하나의 결론밖에 없는 경우 (예: "튕김 = 강제 종료"). 이런 케이스만 묻지 않고 그대로 채운다. 모호하면 **불명**으로 분류해 묻는 쪽으로 기운다.

기존의 `(추정)` 메우기 옵션은 폐기. **불명 한 건이라도 있으면 Step 3로 진입**한다.

### Step 3: 역질문 (`AskUserQuestion` 도구로만)

Step 2에서 **불명** 항목이 1개라도 있거나, 아래 신호가 하나라도 잡히면 반드시 `AskUserQuestion` 도구로 역질문한다. 텍스트로 1, 2, 3 나열하지 않는다.

발동 신호 (해당하면 무조건 묻기):
- **항목/대상**이 광범위해 어떤 화면/기능인지 특정 불가 (예: "버튼이 안 눌림", "게임 시작 누르면 튕김")
- **재현 조건**이 빠져 동일 환경 재현이 불가능 (디바이스/OS/계정/진입 경로 중 어느 하나라도 미상)
- **재현 절차**가 1단계로 환원되지 않거나 빈도가 모호 (예: "가끔", "어쩌다")
- **기대 동작**이 명시 안 됐고 단정적 결론이 어려움
- **버그/개선** 판단이 모호함 (예: "이거 좀 별로임")
- 입력이 10자 미만의 단문 등 정보량이 절대적으로 부족

#### `AskUserQuestion` 사용 규칙

도구 제약: 1회 호출에 1~4개 질문, 각 질문은 2~4개 옵션. "Other"는 자동 추가되므로 사용자가 직접 답할 수 있다.

1. 불명/신호가 잡힌 카테고리를 **최대 4개**까지 추려 한 번에 묻는다 (도구 한도 = 4).
2. 부족한 카테고리가 5개 이상이면 우선순위 높은 4개만 먼저. 답변 후 부족하면 2차 호출.
3. 각 질문의 `header`는 ≤ 12자 (예: "화면", "OS", "빈도", "유형").
4. `options`는 도메인에서 자주 나오는 답 2~4개를 미리 채운다. 단 모든 옵션은 "확정 답"이어야 하며 "잘 모르겠음" 같은 회피 옵션은 넣지 않는다 (회피는 Other로 사용자가 직접 입력).
5. `multiSelect`는 디바이스/OS처럼 복수 답 가능한 카테고리에서만 `true`.

#### 카테고리별 질문 예시 (`references/clarification_questions.md` 참조)

자주 쓰는 질문 패턴:

| 카테고리 | question (예) | header | options (예) |
|---|---|---|---|
| 항목/대상 | "어떤 화면/기능에서 발생/적용되나요?" | "화면" | "타이틀 화면", "메인 로비", "스테이지 진행 중", "상점/인벤토리" |
| 디바이스/OS | "어떤 디바이스/OS에서 발생했나요?" | "OS" | "iOS", "Android", "양쪽 모두", "에디터/PC" |
| 빈도 | "얼마나 자주 발생하나요?" | "빈도" | "매번", "가끔 (절반 이하)", "1~2번만 관찰" |
| 기대 동작 | "원래는 어떻게 동작해야 한다고 생각하셨나요?" | "기대" | 도메인별 보기 2~4개 |
| 유형 | "동작 자체 결함(버그)인가요, 개선 제안인가요?" | "유형" | "버그", "개선" |

#### 답변 처리

1. `AskUserQuestion` 응답이 오면 Step 2로 돌아가 4개 항목을 다시 채운다.
2. 여전히 불명이 남으면 2차 호출. **3차까지는 가능**, 그 이상 왕복은 피한다.
3. 3차 후에도 부족하면 받은 정보로 초안을 만들어 Step 8 확인 단계로 가서 사용자가 직접 수정하도록 위임한다.

### Step 4: 선택(버그/개선) 자동 판단

`references/classification_rules.md`의 §1 룰을 적용한다. 요지:

- **버그**: 기대 동작과 실제 동작이 다름, 오류/충돌/누락/잘못된 값/표시 오류 등 "정상 상태에서 벗어난" 케이스
- **개선**: 동작 자체는 정상이나 UX/편의성/성능/일관성 등을 위한 변경 제안

키워드 신호:
- 버그: "안 됨", "안 보임", "이상함", "오류", "에러", "튕김", "충돌", "잘못된", "누락", "안 들어옴", "꼬임", "표시 안 됨"
- 개선: "추가", "더 좋게", "변경", "수정 제안", "편의성", "직관적", "통일", "개선", "있으면 좋겠"

신호가 섞이면 "정상 동작에서 벗어났는지"를 1차 판단 기준으로 한다.

### Step 5: 우선순위 결정

다음 순서로 결정한다:

1. 사용자가 인자 2번에 `당장`/`상`/`중`/`하` 중 하나를 넘겼다면 그대로 사용
2. 아니면 `references/classification_rules.md` §2의 자동 판단 룰 적용:
   - **당장**: 진행 불가/크래시/결제 손실/데이터 손실 (Blocker)
   - **상**: 핵심 플로우 차단되지 않지만 사용자가 정상 사용 못함 (Critical)
   - **중**: 정상 사용 가능하나 일부 기능 비정상, 또는 다수 사용자 영향 개선 (Major)
   - **하**: 단발성/취향성/사소한 UI/엣지 케이스 (Minor)

판단 근거를 사용자 응답 메시지에 한 줄로 적어 투명하게 한다.

### Step 6: 날짜 결정

`날짜` 컬럼은 입력 기준 날짜(오늘) 사용. ISO-8601 date 형식 (`YYYY-MM-DD`).

오늘 날짜는 시스템 컨텍스트의 `currentDate`에서 얻는다 (예: `2026-05-20`). 인자나 명시적 메시지가 없으면 추가 확인 없이 오늘로 채운다.

### Step 7: 번호(title) 결정 — **다음 정수 자동 부여**

`번호` 컬럼은 데이터소스 내 행의 순번을 나타내는 정수다. 신규 TC는 항상 **(기존 최대 정수 + 1)** 의 title로 가장 마지막에 추가된다.

#### 알고리즘

1. `notion-search`로 데이터소스 내 모든 페이지를 조회한다.
   ```
   notion-search
     query: "" 또는 임의 키워드 (semantic이므로 핵심은 data_source_url 필터)
     data_source_url: 설정의 data_source_url ("collection://..." 형식 그대로)
     page_size: 25 (최대)
     filters: {}
     max_highlight_length: 0
   ```
   응답이 `has_more`이면 `start_cursor`를 사용해 페이지네이션. 모든 페이지를 모은다.
2. 각 결과의 `title`(= `번호` 컬럼 값) 을 정수로 파싱 시도. 정수가 아닌 값(`TC-...` 등 레거시 형식)은 무시.
3. 정수 집합의 최대값 + 1 = 신규 번호. 정수가 하나도 없으면 1.
4. 결과 정수가 문자열로 저장되어 있으므로 신규 번호도 **문자열** 형태로 `번호` 속성에 넣는다 (예: `"6"`).

#### 예외 처리

- `notion-search`가 일시 오류로 실패: 한 번 재시도. 그래도 실패면 사용자에게 안내하고 중단 (임의 번호 부여 금지 — 충돌 위험).
- 같은 번호가 동시 등록 등으로 충돌(노션에서 중복 허용되지만): 등록 후 결과 페이지 title을 한 번 확인. 다른 동시 등록이 같은 정수를 가져갔으면 사용자에게 알린다.
- 사용자가 명시적으로 특정 번호를 지정하길 원하는 경우는 본 스킬 범위 밖. 노션에서 직접 수정하도록 안내.

#### 폐기된 규칙

이전의 `TC-{YYYYMMDD}-{slug}` 명명 규칙은 사용하지 않는다. 설정 파일의 `naming_pattern` 키는 무시된다 (legacy 호환용으로만 남김).

### Step 8: 등록 직전 사용자 확인

노션을 건드리기 전에 작성될 내용을 한 번 보여주고 확인을 받는다:

```
다음 내용으로 노션 TC에 등록하겠습니다:

- 번호: 6 (기존 최대값 5 + 1)
- 날짜: 2026-05-20
- 선택: 버그 (정상 동작에서 벗어남)
- 우선순위: 상 (사용자가 정상 사용 못 함)
- 내용:
  [항목] 보스전 - 캐릭터 궁극기
  [재현 조건] iOS 17, 챕터 5 보스전 진입 후 90초 경과 시점
  [재현 절차]
    1. 보스 체력 50% 이하로 감소
    2. 궁극기 버튼 활성화 확인
    3. 궁극기 버튼 탭
  [기대 동작] 궁극기 애니메이션 재생 + 대미지 발생
  [실제 동작] 버튼은 눌리지만 애니메이션도 대미지도 없음

진행할까요? (예/아니오/수정)
```

`예`면 Step 9로, `아니오`면 중단, `수정`이면 어떤 부분인지 받고 다시 보여준다.

### Step 9: 노션 TC 데이터소스에 페이지 생성

`notion-create-pages`로 단일 페이지를 만든다. 데이터소스 schema (`references/project_config.md` 참조):

```
notion-create-pages
  parent:
    type: "data_source_id"
    data_source_id: 설정의 data_source_id (대시 포함)
  pages:
    - properties:
        번호: "6"   # Step 7에서 계산한 정수의 문자열
        date:날짜:start: "2026-05-20"
        date:날짜:is_datetime: 0
        선택: "버그"
        우선순위: "상"
        내용: "[항목] 보스전 - 캐릭터 궁극기\n[재현 조건] iOS 17, 챕터 5 보스전 진입 후 90초 경과 시점\n..."
        상태: "진행 전"
      content: ""
```

`내용` 컬럼은 text 타입이라 multi-line 가능하지만 마크다운 렌더링이 제한적이라 줄바꿈은 `\n`으로 직접 넣는다. 페이지 본문(content)은 비워둔다 — TC는 행 단위로 관리되고 본문은 안 쓴다.

상태 옵션 이름은 정확히 일치해야 한다: `진행 전`, `진행 중`, `완료`, `처리 불가`. 신규 등록 시에는 항상 `진행 전`.

### Step 10: 결과 보고

성공 보고 포맷:

```
✅ TC 등록 완료

- 페이지: {새 페이지 URL}
- 번호: {번호}
- 선택: {버그|개선} ({선택한 근거 한 줄})
- 우선순위: {상|중|하|당장} ({근거 한 줄})
- 날짜: {YYYY-MM-DD}

노션에서 직접 열어 확인하거나, 담당자가 처리 후 상태를 '진행 중' / '완료'로 변경합니다.
```

실패 시 (notion-create-pages 오류 등)에는 어떤 단계에서 실패했는지, 사용자가 다음에 무엇을 하면 좋은지 안내한다 (예: "데이터소스 권한 확인", "컬럼명이 바뀌었을 가능성").

## 노션 MCP 사용 규칙

- 페이지 생성: `notion-create-pages` (parent.type = `data_source_id`)
- 데이터소스 schema 확인: `notion-fetch` (collection:// URL)
- **기존 행/번호 조회**: `notion-search` (data_source_url = collection://..., page_size=25, filters={}, max_highlight_length=0). `has_more`이면 `start_cursor`로 페이지네이션.
- 페이지 속성 수정 (수정 모드 시): `notion-update-page` (command=`update_properties`)
- TC 외 다른 데이터소스 호출 금지:
  - 기획: `collection://3662dc3e-f514-80eb-afe5-000b7e302c1d`
  - QA 템플릿: `collection://3662dc3e-f514-801c-9d9b-000ba701f95b`

### 데이터소스 ID 입력 형식

- `notion-create-pages`의 `parent.data_source_id`에는 collection 접두 없이 **대시 포함 UUID**를 넣는다.
  - 예: `3662dc3e-f514-809e-aca8-000bbc3d90ba`
- `data_source_url`이 필요한 곳(예: `notion-search`)에는 `collection://` 접두를 그대로 사용.

### Date 타입 컬럼 expanded format

`날짜` 컬럼은 date 타입이므로 expanded property를 써야 한다:

| 키 | 값 |
|----|-----|
| `date:날짜:start` | ISO-8601 (예: `2026-05-20`) |
| `date:날짜:end` | 단일 날짜면 생략 |
| `date:날짜:is_datetime` | `0` (시각 없음) |

`처리 날짜`는 신규 등록 시 비워둔다 (담당자가 처리 완료 시 채움).

## 사용자 확인이 필요한 시점

- 최초 프로젝트 설정 시 데이터소스 URL 확인
- TC 내용에 불명/애매한 부분이 1개라도 있는 경우 → `AskUserQuestion` 도구로 역질문 (Step 3)
- 데이터소스 schema가 설정 파일과 불일치할 때 매핑 확인
- 등록 직전 최종 확인 (Step 8)

## 절대 하지 말 것

- 사용자에게 묻지 않고 추측으로 빈칸을 메워 등록하기 (`(추정)` 메우기 금지)
- 역질문을 텍스트 1, 2, 3 나열로 하기 — 반드시 `AskUserQuestion` 도구 사용
- `번호`에 `TC-...-슬러그` 등 비정수 값을 부여하기 (정수 + 1 규칙 위반)
- 기존 번호 조회 없이 임의 정수를 할당하기 (충돌 위험)
- TC 외 데이터소스(기획/QA 템플릿) 읽기/수정
- 페이지 본문에 코드/구현 작성
- 선택(버그/개선)을 사용자에게 묻기 (자동 판단해야 함)
- 우선순위가 명시되었는데 자동 판단으로 덮어쓰기
- 노션 상태를 `진행 전` 외 값으로 신규 등록
- 같은 TC 내용을 중복 등록 (사용자가 명시 요청한 경우만 허용)

## 이 스킬 자체를 업데이트하려면

`references/maintainer.md`를 먼저 읽는다. 핵심 요점:

- 스킬 파일은 사용자의 로컬 작업본(예: `/Users/<id>/geuneda-plugins/plugins/tc-dev-writer/`)에 있다. 직접 수정 후 commit + push 하면 팀원이 `/plugin marketplace update geuneda-plugins`로 받아간다. (이전엔 `~/.claude/plugins/marketplaces/geuneda-plugins/...` 경로를 사용하라고 안내했으나, 실제로는 사용자별 작업본 위치를 우선한다.)
- 노션 TC 데이터소스 schema가 변경되면 `references/project_config.md`의 매핑 안내를 같이 갱신한다.
