# 기획서 검토 체크리스트

기획서를 개발 명세서로 변환하기 전 검토하는 기준. **기획자가 결정해야 할 것**만 검토 필요로 분류하고, **개발자가 알아서 결정/추론할 수 있는 것**은 추론 항목으로 명세서에 채운다.

## 검토의 첫 번째 원칙

> 기획자에게 묻지 않아도 답이 나오는 것은 검토 필요가 아니다.

기획자의 시간을 아끼고 의미 있는 결정에만 집중하게 만든다. 잘못된 검토 요청은 기획자가 "당연한 거 아니야?"라고 느끼게 만들어 협업 신뢰를 깎는다.

## 검토 필요로 처리하지 않는 것 (먼저 화이트리스트)

다음은 모두 명세서에 `(추론)` 표시와 함께 채우거나, 자연스러운 표준 처리를 적용한다. **절대 검토 필요로 처리하지 않는다.**

### 1. 시트로 관리될 데이터 항목

기획 데이터(밸런싱/콘텐츠/가격)는 구글 시트로 관리되는 것이 일반적이다. 다음은 검토 항목이 아니다:

- 확률표 (가챠 등급별 확률, 드롭률)
- 가격표 (상품/박스 가격, 재화 환율)
- 보상 풀 (아이템 종류와 등급)
- 콘텐츠 데이터 (스테이지 수치, 적 능력치, 무기 데미지)
- 박스/아이템 정의 테이블의 디테일 (이름, 아이콘 키, 등급 등)

명세서에는 "이 데이터는 구글 시트 `{시트명}`으로 관리됨"으로 적고, 시트가 아직 없으면 "추후 시트 생성 예정" 표시.

### 2. 저장 위치 (프로젝트 패턴에 따라 자동 결정)

`.plan-to-devspec.json`의 `storage_pattern` 값을 본다:

- `"server"` (서버 기반 프로젝트): 영구 데이터는 서버 DB
- `"local"` (로컬 저장 기반): 영구 데이터는 로컬 (PlayerPrefs/SQLite/JSON)
- 둘 다 명시 안 됨: 사용자에게 한 번만 묻고 설정 파일에 저장

추가 분류:
- 초기화돼도 되는 설정/UI 상태: 캐시 (메모리 또는 임시 파일)
- 보안/검증 필요한 데이터 (결제, 재화, 보상): 항상 서버

이 룰로 결정 가능한 저장 위치는 검토 항목이 아니다. 명세서에 `(추론: 프로젝트 패턴 server에 따라 서버 저장)` 표시.

### 3. 데이터베이스 스키마 디테일

테이블명/필드명/타입/PK/인덱스 같은 구현 세부는 개발자 영역이다. 기획서가 "구매 이력을 저장한다"고만 적으면 충분하다. 스키마는 명세서에서 개발자가 정의한다.

### 4. 표준 UX / 자연스러운 처리

다음은 게임 개발에서 표준 관행이므로 명세서에 추론으로 채운다:

- 로딩 인디케이터 / 에러 토스트
- 결제/보상 수령 시 멱등성 처리 (중복 결제 방지)
- 네트워크 실패 시 재시도 또는 명확한 에러 안내
- 백그라운드 복귀 시 상태 동기화
- 버튼 잠금 (처리 중 다중 입력 방지)
- 인벤토리 가득 시 우편함 보관 (게임에 우편함이 있다면)
- 강제 종료 후 복원 (서버 권위 데이터 기반)

### 5. 개발 디테일 (코드 구조, 클래스, API)

기획자 영역 아님. 명세서에서 개발자가 정의.

## 검토 필요 판단 기준 (이건 진짜 물어야 한다)

기획자만 답할 수 있는 것. 다음 중 하나라도 해당하면 검토 필요로 처리한다.

### A. 모호한 표현 (측정 불가)

기획자가 의도한 구체값을 모르면 진행 불가능한 표현. 단순 수치는 시트로 빠지므로(위 §1) 카테고리에서 제외하고, **기획 의도 자체가 모호한 경우만** 다룬다.

판단 기준: "이 표현을 구체화하지 않으면 명세서가 작성 불가능한가?"

- "유저가 충분히 매력적이라고 느낄" → 무엇이 매력의 기준인가
- "재미있는 연출" → 어떤 종류의 재미인가 (긴장감/유머/임팩트)
- "직관적인 UI" → 무엇 기준의 직관

단, 시트로 빠질 수치 표현("적당한 확률", "저렴한 가격")은 시트 관리 항목으로 분류하고 검토 필요로 처리하지 않는다.

### B. 비즈니스 정책 결정

비즈니스 임팩트가 큰데 기획서에 정해지지 않은 정책.

- 환불/이의제기 대응 정책 (지급된 보상 회수 여부)
- 사용량 한도 (일일/계정 누적 구매 한도)
- 천장(피티) 시스템 적용 여부 + 기준 횟수
- 만료/소멸 정책 (이벤트 종료 후 미수령 보상 처리)
- 어뷰징 방지 정책 (자기 자신 초대, 멀티 계정 등에서 어디까지 허용)

### C. 핵심 동작 규칙 누락

조건 분기에서 어느 쪽이 의도된 동작인지 기획자만 답할 수 있는 경우.

- 보상 수령이 자동 vs 수동인가
- 박스 구매 시 즉시 오픈 vs 인벤토리 보유 후 나중 오픈인가
- 이벤트 기간 종료 시점에 진행 중인 동작은 무효 vs 유효인가
- "튜토리얼 완료"의 정확한 기준 (특정 퍼널 클리어 / N레벨 도달 등)

표준 UX로 추론 가능한 동작(§4)은 여기 포함되지 않는다.

### D. UI/UX 의도 또는 레퍼런스 부재

UI 의도를 알아야 명세 가능한데 기획서에 텍스트 설명만 있고 의도가 불명확한 경우.

- "직관적인 메뉴 배치" → 어떤 정보 우선순위?
- "임팩트 있는 연출" → 톤이 진지/유쾌/긴장?
- 신규 화면인데 레이아웃 레퍼런스나 와이어프레임이 없음 → 비슷한 기능의 레퍼런스 게임 스크린샷이라도 요청

요청 시 기획자가 이해하기 쉬운 표현으로:

> "{화면}이 어떤 느낌이었으면 좋겠는지 참고 이미지 1~2장 첨부해 주실 수 있을까요? 다른 게임 스크린샷도 좋습니다."

표준 처리(§4) 가능한 UI 항목(로딩 인디케이터 등)은 여기 포함되지 않는다.

### E. 시스템 연동 의도

연동되는 기존 시스템을 어떻게 다룰지 기획자 결정이 필요한 경우.

- 기존 보상 시스템과 신규 보상 시스템의 우선순위
- 영향받는 기존 기능이 기획자 의도에 맞는 변경인지 확인
- 마이그레이션 시 기존 유저 데이터 처리 (예: 기존 누적값 0으로 리셋 vs 유지)

### F. 기획서 내 충돌/모순

명백한 모순. 둘 중 어느 쪽이 의도인지 묻는다.

- A 섹션 "최대 5회" vs B 섹션 "6회차 보상 정의"
- 같은 보상의 수량이 두 곳에서 다름
- 명시된 규칙 간 산술이 안 맞음

## 검토 필요 처리 절차

1. 위 카테고리 중 해당 항목 발견 시, 다음 형식으로 비고에 요청사항을 작성한다. **반드시 기획자가 이해하기 쉬운 표현**으로.

```
[검토 필요] 다음 항목들에 대한 결정/추가 정보가 필요합니다:

[카테고리명: 한글로]
- {원문 인용 또는 구체 항목}
  → {왜 필요한지 / 무엇을 원하는지}

(해당하는 카테고리만)

위 항목들이 결정되어야 개발 명세서 작성이 가능합니다.
```

2. 노션 상태를 `검토 필요`로 변경하고 비고를 위 텍스트로 덮어쓴다.
3. 기존 비고에 `[검토 필요]` 로 시작하지 않는 의미 있는 메모가 있었다면 새 비고 마지막에 `> 이전 메모: {원문 한 줄}` 형식으로 보존한다.
4. 사용자에게 결과를 요약 보고하고 **즉시 중단**한다.

## 비고 작성 톤

- 비난하지 않는 톤 ("X가 없네요" 보다 "X를 명시해 주실 수 있을까요?")
- 개발 용어는 최소화 (`스키마`, `테이블`, `엔드포인트` 같은 단어 피하기)
- 보기 옵션이 가능하면 함께 제시 ("자동인가요, 아니면 유저가 직접 받기 버튼을 눌러야 하나요?")
- 원문은 따옴표로 인용해 어떤 부분 얘기인지 명확히

## 실제 예시 (개선된 버전)

기획서 "랜덤 박스 가챠"에 대한 검토 (시트 관리/저장 위치/표준 UX는 제외, 비즈니스/UI 의도/모호 표현만 남음):

```
[검토 필요] 다음 항목들에 대한 결정/추가 정보가 필요합니다:

모호한 표현 (기획 의도 구체화 필요)
- "유저가 충분히 매력적이라고 느낄 만한 보상 구성"
  → "매력적"의 의도가 어느 방향인가요? 예를 들어 "10번 열면 1번은 SR 이상이 나온다" 같은 체감 기준 또는 비교 게임 사례가 있으면 도움이 됩니다.

비즈니스 정책 결정 필요
- 천장(피티) 시스템을 적용하나요? 적용한다면 몇 회까지가 기준이고, 어떤 등급을 보장하나요?
- 박스별 1일 또는 계정 누적 구매 한도가 있나요? 없다면 무제한으로 두어도 되나요?
- 환불/이의제기 발생 시 이미 지급된 박스 보상을 회수하나요?

핵심 동작 규칙 누락
- 박스를 구매하면 바로 열리나요, 아니면 인벤토리에 보관하다가 유저가 직접 열기를 누르나요?
- "10연차" 같은 일괄 구매를 지원하나요?

UI/UX 의도 (레퍼런스 요청)
- "박스 오픈 시 화려한 연출"의 톤이 어떤 느낌이면 좋을까요? 비슷한 분위기의 다른 게임 박스 오픈 영상이나 스크린샷 1~2개 첨부해 주시면 명세에 반영하기 좋습니다.

위 항목들이 결정되어야 개발 명세서 작성이 가능합니다.

(참고: 박스 종류별 확률표/가격/보상 풀은 구글 시트로 관리할 예정이므로 이 검토에서는 제외했습니다.)
```

이전 버전과 비교해 검출 건수는 줄었지만, 기획자가 답해야만 명세 가능한 진짜 결정 사항만 남았다.
