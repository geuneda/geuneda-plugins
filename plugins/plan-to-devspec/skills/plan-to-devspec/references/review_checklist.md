# 기획서 검토 체크리스트

기획서를 개발 명세서로 변환하기 전에 점검하는 항목. 하나라도 걸리면 **검토 필요** 상태로 표시하고 비고에 요청사항을 작성한 뒤 즉시 중단한다.

## 검토 기준 카테고리

### A. 모호한 표현
구체적 수치나 조건으로 치환 불가능한 표현이 있는가?
- "적당히", "충분히", "많이", "빠르게", "자연스럽게"
- "유저가 좋아할 만한", "재미있는", "직관적인"
- 단위 없는 숫자 ("쿨다운 3" -> 초인지 프레임인지 불명)

### B. 데이터 정의 누락
- 신규 데이터 필드의 타입/범위/기본값이 명시되지 않음
- 저장 위치(서버/로컬)가 불명확
- 직렬화 방식이 정해지지 않은 영구 데이터

### C. 동작 규칙 공백
- 조건 분기에서 한쪽 케이스 처리가 빠짐 (성공만 있고 실패 없음)
- 시작/종료 조건만 있고 중간 상태 전이 규칙 없음
- "재진입 시 어떻게 동작하는가" 같은 흔한 케이스 누락

### D. UI/UX 갭
- 화면은 언급되지만 진입/종료 경로가 불명
- 상태 표시(Loading/Empty/Error)가 정의되지 않음
- 인터랙션 결과 피드백 누락 (성공/실패 시 무엇이 보이는가)

### E. 시스템 연동 위험
- 영향받는 기존 시스템이 명시되지 않았는데 변경 영향이 명백
- 다른 작업 순서 항목과 의존 관계가 있는데 순서가 어긋남
- 마이그레이션/하위 호환성 처리 미언급

### F. 예외 케이스 누락
- 네트워크 실패, 동시 입력, 중복 호출 등 흔한 예외 케이스 처리 없음
- 데이터 부재/손상 시 동작 없음
- 백그라운드/포그라운드 전환 시 동작 없음

### G. 충돌/모순
- 한 곳에서는 X라 하고 다른 곳에서는 Y라고 서술
- 명시된 수치들이 산술적으로 안 맞음 (예: 최대 5인데 6번째 보상 정의)

## 검토 필요 처리 절차

1. 위 카테고리 중 하나라도 해당하는 항목 발견 시, 다음 형식으로 비고에 요청사항을 작성한다:

```
[검토 필요] 다음 항목들을 명확히 해주세요:

A. 모호한 표현 (구체 수치 필요)
- "{원문 인용}" -> {구체적으로 무엇이 필요한지}

B. 데이터 정의 누락
- {필드명/항목명} 의 타입/저장 위치/기본값 명시 필요

C. 동작 규칙 공백
- {조건} 일 때의 처리 방식 미정

D. UI/UX 부가 정보
- {요소} 의 {요청 사항}

E. 시스템 연동 위험
- {시스템} 과의 {연동/마이그레이션} 명시 필요

F. 예외 케이스 누락
- {상황} 에 대한 처리 방식 미정

G. 충돌/모순
- {위치1} 의 {값1} vs {위치2} 의 {값2} — 일관성 확인 필요

(해당하는 카테고리만 작성. 한 카테고리 안에 여러 항목이면 줄바꿈으로 나열)

위 항목들이 결정되어야 개발 명세서 작성이 가능합니다.
```

### 실제 작성 예시 (랜덤 박스 가챠 - 모호 표현/데이터 누락 검출)

```
[검토 필요] 다음 항목들을 명확히 해주세요:

A. 모호한 표현 (구체 수치 필요)
- "각 박스마다 적당한 확률" -> 박스별 등급별 확률표 필요 (예: 일반 박스 R 60%, SR 30%, SSR 10%)
- "자주", "가끔" -> 수치로 교체
- "충분히 매력적이라고 느낄 만한" -> 측정 불가, 보상 구성 구체화 필요
- "저렴한", "중간", "비싸지만" -> 박스별 구매 가격(재화 종류 및 수량) 명시

B. 데이터 정의 누락
- 박스별 확률표 저장 위치 (서버/클라이언트 정적 데이터) 명시 필요
- 구매 이력/보유 수 테이블 스키마 미정의
- 구매 시 사용하는 재화 종류(골드/보석/현금) 명시 필요

C. 동작 규칙 공백
- 박스 구매 시 즉시 오픈인가요, 아니면 인벤토리에 보유 후 나중 오픈인가요?
- 동일 박스 다수 보유/일괄 구매/10연차 등 지원 여부?
- 천장(피티) 시스템 적용 여부? 적용 시 기준 횟수?
- 인벤토리가 가득 차서 보상 지급 불가 시 처리 방식?

F. 예외 케이스 누락
- 박스 오픈 중 네트워크 실패 시 처리 (이미 결제됐다면 재시도, 결제 전이면 취소)
- 중복 결제 방지 방식 (서버 멱등성 처리)
- 환불/이의제기 흐름 (CS 대응 기준)

위 항목들이 결정되어야 개발 명세서 작성이 가능합니다.
```

2. 노션 상태를 `검토 필요`로 변경하고 비고를 위 텍스트로 덮어쓴다.
3. 기존 비고에 `[검토 필요]` 로 시작하지 않는 의미 있는 메모가 있었다면 새 비고 마지막에 `> 이전 메모: {원문 한 줄}` 형식으로 보존한다.
4. 사용자에게 결과를 요약 보고하고 **즉시 중단**한다. 다음 작업 순서로 넘어가지 않는다.

## 비고 작성 톤

- 비난하지 않는 톤으로 작성한다 ("X가 없네요" 보다 "X를 명시해 주세요")
- 한 항목당 한 줄로 간결하게
- 가능하면 보기 옵션을 제시 ("초 단위인가요, 프레임인가요?")
- 원문 인용은 따옴표로 정확히

## 검토 필요로 처리하지 않는 경우

다음은 검토 필요가 아니다. 추론하여 명세서에 `(추론)` 표시와 함께 채운다:
- 명백한 관례적 처리 (예: 버튼 비활성화 시 회색 처리)
- 게임 전반의 공통 규칙으로 이미 결정된 사항
- 기획서가 `[추후]`/`[보류]` 태그로 명시한 미래 작업 항목
- 코드 구조나 클래스 설계 등 구현 디테일 (기획자 결정 영역 아님)
