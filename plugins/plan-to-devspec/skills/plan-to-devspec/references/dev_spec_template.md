# 개발 명세서 템플릿

기획서를 분석해 개발자가 즉시 구현할 수 있도록 작성하는 명세서 템플릿이다. 모호한 표현 없이 구체적 수치, 조건, 규칙으로 채운다.

## 파일 위치 규칙

- 저장 경로: `docs/dev-specs/{기획서_제목_slug}.md`
- slug 변환 규칙:
  - 공백 -> `-`
  - 특수문자 제거
  - 한글 유지 가능
  - 예: `루팅 시스템` -> `루팅-시스템.md`
- 같은 이름이 이미 존재하면 사용자에게 덮어쓸지 묻는다

## 문서 템플릿

```markdown
# {기획서 제목} 개발 명세서

> 원본 기획서: {노션 페이지 URL}
> 기획 상태: 기획 완료 -> 개발 진행중
> 작성일: {YYYY-MM-DD}
> 작업 순서: {N}

---

## 1. 기능 요약

- **목적**: 한 줄 요약 (왜 만드는가)
- **핵심 동작**: 두세 줄로 무엇을 하는지
- **선행 조건**: 이 기능 동작에 필요한 전제 조건

---

## 2. 구현 범위

### 2.1 포함 (In Scope)
- [ ] 항목 1
- [ ] 항목 2
- [ ] 항목 3

### 2.2 제외 (Out of Scope)
- 항목 4 (사유: ...)
- 항목 5 (사유: 추후 단계)

---

## 3. 데이터 모델

### 3.1 신규 데이터 구조
| 필드명 | 타입 | 설명 | 기본값 | 비고 |
|--------|------|------|--------|------|
| field_a | int | ... | 0 | ... |
| field_b | string | ... | "" | optional |

### 3.2 영향받는 기존 데이터
- `ExistingTable.field_x`: ... 추가 의미
- 마이그레이션 필요 여부: 예/아니오

### 3.3 직렬화/저장
- 저장 위치 (로컬/서버/Addressables 등)
- 직렬화 형식 (JSON/PlayerPrefs/ScriptableObject)

---

## 4. 비즈니스 로직

각 규칙은 `When X, do Y` 형태로 검증 가능하게 작성한다.

### 4.1 핵심 규칙
1. **R1**: When `<조건>`, then `<결과>`.
2. **R2**: ...

### 4.2 수치/밸런스
| 항목 | 값 | 출처 |
|------|----|----|
| 쿨다운 | 3s | 기획서 §X |
| 최대 누적 | 5 | 기획서 §Y |

### 4.3 상태 전이
- 상태 목록: `Idle` / `Active` / `Cooldown` / `Disabled`
- 전이 규칙: `Idle -> Active (입력 발생)`, `Active -> Cooldown (실행 완료)` ...

---

## 5. UI/UX

### 5.1 화면/뷰
- 신규 화면: `{화면명}` - 진입 경로, 표시 정보, 닫힘 조건
- 수정되는 화면: `{화면명}` - 변경점

### 5.2 인터랙션
- 버튼/제스처 별 동작
- 피드백 (애니메이션/사운드/햅틱)

### 5.3 상태별 표시
| 상태 | 보이는 것 | 비활성화 요소 |
|------|-----------|---------------|
| Idle | ... | ... |

---

## 6. 의존성 / 영향 시스템

- **읽어야 할 시스템**: ...
- **수정해야 할 시스템**: ...
- **외부 패키지**: Geuneda.UIService, Addressables 등
- **다른 작업 순서와의 관계**: 작업 N에서 만든 X를 사용

- [ ] T1: 데이터 모델 정의

## 10. 기획서와의 매핑

| 명세서 섹션 | 기획서 출처 |
|-------------|-------------|
| §1 기능 요약 | 기획서 §1 |
| §3 데이터 모델 | 기획서 §5 또는 추론 |
| §4.2 수치 | 기획서 §3 |

> 기획서에 없어 명세서 작성 중 결정/추론한 항목은 `(추론)` 으로 명시한다.
```

## 작성 원칙

- **수치는 반드시 구체값**: "적당히", "충분히" 같은 표현 금지
- **조건은 반드시 검증 가능한 형태**: "잘 동작" 같은 표현 금지
- **추론한 부분은 명시**: 기획서에 없어 작성자가 채운 항목은 `(추론)` 표시
- **개발 외 요소 배제**: 일정, 인력 추정, 마케팅 카피 등은 포함하지 않음
- **상태 태그 통합**: 기획서가 `[미정]`/`[보류]` 등 태그를 가졌다면 그대로 명세서에 반영
