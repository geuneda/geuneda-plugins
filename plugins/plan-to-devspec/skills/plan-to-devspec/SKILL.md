# 기획서 -> 개발 명세서 변환 스킬

노션 기획 데이터베이스를 작업 순서대로 처리해 개발 가능한 마크다운 명세서를 만든다. 기획서에 모호한 부분이 있으면 검토 요청으로 돌리고 멈춘다.

## 핵심 원칙

- **작업 순서를 절대 지킨다**: `작업 순서` 컬럼의 오름차순으로 처리한다. 건너뛰지 않는다.
- **검토 필요는 즉시 중단**: 한 항목이라도 검토 필요 처리되면 다음 항목으로 넘어가지 않는다.
- **기획자 영역과 개발자 영역을 엄격히 구분한다**: 기획자에게는 **비즈니스 정책/의도/UI 의도**만 묻는다. 시트로 관리될 데이터(확률표/가격/보상 풀), 저장 위치, DB 스키마 디테일, 표준 UX(로딩/에러/재시도/멱등성)는 기획자에게 묻지 않고 `(추론)` 표시와 함께 명세서에 채운다. (`references/review_checklist.md` 참조)
- **다른 데이터 소스 절대 미수정**: TC/QA 등 기획이 아닌 데이터 소스는 읽지도 쓰지도 않는다.
- **기획자에게 친화적인 표현**: 검토 비고는 개발 용어(스키마/엔드포인트 등) 최소화. 의도와 목적을 명확히 전달.

## 워크플로우

### Step 1: 프로젝트 설정 확인

`references/project_config.md`에 따라 데이터 소스 URL과 컬럼 매핑을 확보한다.

1. 현재 작업 디렉토리에서 `./.plan-to-devspec.json` 확인
2. 없으면 `~/.claude/plan-to-devspec.config.json` 확인
3. 둘 다 없으면 사용자에게 데이터베이스/데이터 소스 URL을 묻고 새로 작성

설정 로드 후 `output_dir`(기본 `docs/dev-specs`) 경로가 없으면 `mkdir -p`로 생성한다.

### Step 2: 처리 대상 페이지 목록 조회

노션 MCP에는 `query_data_sources` 도구가 없으므로 `notion-search`를 사용한다.

```
notion-search:
  query: "시스템 보상"     # 일반적인 단어. 데이터 소스 내 페이지를 폭넓게 수집할 수 있는 문구
  data_source_url: 설정의 data_source_url
  page_size: 25
  max_highlight_length: 0
  filters: {}
```

받은 결과(`{id, title, url}` 목록)에 대해 각 페이지를 `notion-fetch`로 한 번씩 조회해 `properties.상태`, `properties.작업 순서` 값을 확보한다. 페이지가 25개를 초과하면 더 일반적인 query로 한 번 더 검색하거나 사용자에게 페이지 URL 목록을 직접 받는다.

수집한 목록에서:
- `상태 == "기획 완료"` 인 항목만 남긴다
- `작업 순서` 오름차순으로 정렬한다
- 동일 `작업 순서` 값이 있으면 페이지 생성 시각(createdTime) 오름차순으로 2차 정렬

처리할 항목이 없으면 사용자에게 보고하고 종료한다.

처리 직전 사용자에게 처리 대상 목록(작업 순서, 제목)을 요약 표로 보여주고 진행을 시작한다. (확인 질문은 하지 않는다 — 단순 통보)

### Step 3: 다음 항목 처리 루프

가장 작은 작업 순서 항목부터 한 건씩 다음 절차를 반복한다.

#### 3-1. 기획서 페이지 읽기

해당 항목의 페이지를 `notion-fetch`로 읽어 본문 마크다운을 확보한다.

페이지 본문이 비어 있거나(섹션 헤더가 하나도 없는 수준) 접근 불가하면:
- 검토 필요로 처리하고 비고에 "기획서 본문이 비어 있어 검토할 수 없습니다. 기획 내용을 채워주세요." 작성
- **즉시 중단**

#### 3-2. 기획서 검토

`references/review_checklist.md`의 카테고리 A~G를 점검한다. 각 카테고리는 객관적 판단 기준이 있으니 임의로 완화하지 않는다.

검토 결과 분기:
- **통과**: Step 3-3로 진행
- **검토 필요**: Step 3-4로 진행

#### 3-3. 통과 -> 개발 명세서 작성

`references/dev_spec_template.md`의 템플릿을 사용해 마크다운을 작성한다.

저장:
- 경로: 설정의 `output_dir` (디렉토리 없으면 미리 mkdir)
- 파일명: `{slug(기획서_제목)}.md`
  - slug 규칙: 공백 -> `-`, 특수문자 제거, 한글 유지
  - 예: "일일 출석 보상 시스템" -> `일일-출석-보상-시스템.md`
- 같은 파일이 존재하면 사용자에게 덮어쓸지 확인

명세서 작성 후 노션 페이지 업데이트:

```
notion-update-page:
  page_id: 페이지 ID
  command: "update_properties"
  properties: { "상태": "개발 진행중" }
  content_updates: []
  position: { "type": "end" }
```

상태만 변경한다. `비고`는 손대지 않는다 (기존 메모 보존).

성공 보고:
- 생성된 명세서 절대 경로
- 변경된 노션 페이지 URL

다음 항목으로 진행.

#### 3-4. 검토 필요 -> 노션 업데이트 후 중단

`references/review_checklist.md`의 비고 작성 포맷에 따라 요청 텍스트를 만든다.

노션 페이지 업데이트:

```
notion-update-page:
  page_id: 페이지 ID
  command: "update_properties"
  properties:
    상태: "검토 필요"
    비고: "[검토 필요] ... (체크리스트 포맷)"
  content_updates: []
  position: { "type": "end" }
```

기존 비고에 의미 있는 텍스트(`[검토 필요]` 로 시작하지 않는 메모)가 있었다면 새 비고 마지막에 `> 이전 메모: {원문}` 한 줄로 보존한다.

사용자에게 다음을 보고한다:
```
[검토 필요] 작업 순서 {N}: "{기획서 제목}"
페이지: {URL}

검출된 항목:
- A. 모호한 표현: 항목 수
- B. 데이터 정의 누락: 항목 수
- ...

비고 전문은 노션 페이지에서 확인 가능합니다.
기획자 답변 후 상태를 다시 '기획 완료'로 변경한 뒤 스킬을 재실행해 주세요.
```

**다음 항목으로 절대 진행하지 않는다. 종료한다.**

### Step 4: 완료 보고

모든 항목 처리 또는 검토 필요로 중단 시:

- 처리 완료된 항목 수와 생성된 명세서 파일 경로 목록
- 검토 필요로 변경된 항목 (있다면 페이지 URL)
- 처리 안 한 항목 (검토 필요 중단 이후 남은 항목 수)
- 다음 액션 안내

## 노션 MCP 사용 규칙

- 데이터 소스 페이지 목록 조회: `notion-search` (data_source_url 지정)
- 데이터 소스/페이지 스키마: `notion-fetch` (collection:// 또는 페이지 URL/ID)
- 페이지 속성 업데이트: `notion-update-page` (command=`update_properties`)
- 기획 데이터 소스 외 다른 데이터 소스(TC: `collection://3662dc3e-f514-809e-...`, QA: `collection://3662dc3e-f514-801c-...`)는 절대 호출하지 않는다.

### notion-update-page 필수 파라미터

`update_properties` 명령이라도 다음 파라미터가 모두 필요하다:

| 파라미터 | 값 |
|---------|----|
| `page_id` | 페이지 ID |
| `command` | `"update_properties"` |
| `properties` | 업데이트할 속성 JSON |
| `content_updates` | `[]` (빈 배열) |
| `position` | `{"type": "end"}` (사용 안 해도 필수) |

생략하면 검증 오류가 발생한다.

### Status 타입 컬럼 옵션 이름

상태 컬럼은 `status` 타입이다. 띄어쓰기 포함 정확히 일치시켜야 한다:
- `기획 전`, `기획 완료`, `검토 필요`, `개발 진행중`, `개발 완료`

## 사용자 확인이 필요한 시점

- 최초 프로젝트 설정 시 데이터 소스 URL 확인
- 같은 파일명이 이미 존재할 때 덮어쓰기 여부
- 데이터 소스 컬럼명/상태명이 일치하지 않을 때 매핑 확인
- 페이지가 25개 초과해 페이지네이션이 필요한 경우 추가 query 또는 URL 목록 직접 받기

## 절대 하지 말 것

- 작업 순서 무시하고 건너뛰기
- 검토 필요로 처리하고도 다음 항목 진행
- 기획 외 데이터 소스(TC/QA 등) 읽기/수정
- 기획자가 결정해야 할 사항을 임의로 결정 (수치, 규칙, 우선순위 등)
- 노션 페이지를 새로 만들기 (기존 페이지 업데이트만 한다)
- 명세서에 코드 예시/구현 코드 작성 (명세서는 명세이지 구현이 아님)

## 참조 파일

- `references/project_config.md` - 프로젝트별 설정 파일 포맷과 신규 설정 절차
- `references/review_checklist.md` - 검토 필요 판단 기준과 비고 작성 포맷
- `references/dev_spec_template.md` - 개발 명세서 마크다운 템플릿과 작성 원칙
- `references/maintainer.md` - 이 스킬 자체를 수정/배포하는 가이드 (파일 위치, 동기화, 테스트, push)

## 이 스킬 자체를 업데이트하려면

스킬 코드를 고치거나 새 기능을 더하려면 `references/maintainer.md`를 먼저 읽는다. 핵심 요점:

- 스킬 파일은 두 곳에 존재: 로컬(`~/.claude/skills/plan-to-devspec/`)과 마켓플레이스 git 클론(`~/Desktop/geuneda-plugins/plugins/plan-to-devspec/skills/plan-to-devspec/`). 변경 시 양쪽 동기화 필수.
- 테스트 디렉토리(`~/Desktop/plan-to-devspec-test/`)와 노션 테스트 데이터(데이터 소스 + 6개 시나리오)가 미리 준비되어 있다.
- 변경 후 `~/Desktop/geuneda-plugins`에서 git commit + push 하면 팀원이 `/plugin marketplace update geuneda-plugins`로 받아감.
