# 기획서 -> 개발 명세서 변환 스킬

노션 기획 데이터베이스를 작업 순서대로 처리해 개발 가능한 마크다운 명세서를 만든다. 기획서에 모호한 부분이 있으면 검토 요청으로 돌리고 멈춘다.

## 핵심 원칙

- **작업 순서를 절대 지킨다**: `작업 순서` 컬럼의 오름차순으로 처리한다. 건너뛰지 않는다.
- **검토 필요는 즉시 중단**: 한 항목이라도 검토 필요 처리되면 다음 항목으로 넘어가지 않는다.
- **기획자 영역과 개발자 영역을 구분한다**: 수치/규칙/조건은 기획자에게 묻고, 코드 구조는 추론한다.
- **다른 데이터 소스 절대 미수정**: TC/QA 등 기획이 아닌 데이터 소스는 읽지도 쓰지도 않는다.

## 워크플로우

### Step 1: 프로젝트 설정 확인

`references/project_config.md`에 따라 데이터 소스 URL과 컬럼 매핑을 확보한다.

1. 현재 작업 디렉토리에서 `./.plan-to-devspec.json` 확인
2. 없으면 `~/.claude/plan-to-devspec.config.json` 확인
3. 둘 다 없으면 사용자에게 데이터베이스/데이터 소스 URL을 묻고 새로 작성

### Step 2: 처리 대상 조회

설정된 데이터 소스를 `notion-fetch` 또는 `notion-search`로 조회해 다음을 가져온다:

- 상태가 **`기획 완료`** 인 항목만 (기획 전/검토 필요/개발 진행중/개발 완료는 제외)
- `작업 순서` 오름차순 정렬
- 항목별 페이지 URL, 기획서 제목, 작업 순서

처리할 항목이 없으면 사용자에게 보고하고 종료한다.

### Step 3: 다음 항목 처리 루프

가장 작은 작업 순서 항목부터 한 건씩 다음 절차를 반복한다.

#### 3-1. 기획서 페이지 읽기

해당 항목의 `기획서` 컬럼이 가리키는 노션 페이지를 `notion-fetch`로 읽는다.

페이지가 비어 있거나 접근 불가하면:
- 검토 필요로 처리하고 비고에 "기획서 페이지를 읽을 수 없습니다" 작성
- **즉시 중단**

#### 3-2. 기획서 검토

`references/review_checklist.md`의 카테고리 A~G를 점검한다.

검토 결과 분기:
- **통과**: Step 3-3로 진행
- **검토 필요**: Step 3-4로 진행

#### 3-3. 통과 -> 개발 명세서 작성

`references/dev_spec_template.md`의 템플릿을 사용해 마크다운을 작성한다.

저장:
- 경로: 설정의 `output_dir` 기준 (기본 `docs/dev-specs/`)
- 파일명: `{기획서_제목}.md` (slug 규칙은 템플릿 참조)
- 같은 파일이 존재하면 사용자에게 덮어쓸지 확인

명세서 작성 후 노션 페이지 업데이트:
- `상태` -> `개발 진행중`
- `비고`는 변경하지 않음 (기존 메모 보존)

성공 보고 후 다음 항목으로 진행.

#### 3-4. 검토 필요 -> 노션 업데이트 후 중단

`references/review_checklist.md`의 비고 작성 포맷에 따라 요청 텍스트를 만든다.

노션 페이지 업데이트:
- `상태` -> `검토 필요`
- `비고` -> 요청 텍스트 덮어쓰기 (기존 내용은 검토 요청 텍스트 위에 보존하지 않음. 단, 기존 비고가 있다면 마지막에 `> 이전 메모: ...` 형식으로 한 줄 보존)

사용자에게 다음을 보고한다:
```
[검토 필요] 작업 순서 {N}: "{기획서 제목}"

다음 항목들에 명확한 답변이 필요해 처리를 중단합니다:
- {요청사항 요약}

기획자 답변을 받으면 해당 페이지의 상태를 다시 '기획 완료'로 변경한 뒤 스킬을 재실행해 주세요.
```

**다음 항목으로 절대 진행하지 않는다. 종료한다.**

### Step 4: 완료 보고

모든 항목이 처리되었거나 검토 필요로 중단되었을 때 다음을 사용자에게 보고한다:

- 처리 완료된 항목 수와 생성된 명세서 파일 목록
- 검토 필요로 변경된 항목 (있다면)
- 다음 액션 안내 (검토 회신 후 재실행 등)

## 노션 MCP 사용 규칙

- 데이터 소스 조회: `notion-fetch`에 `collection://...` URL 전달
- 페이지 내용 읽기: `notion-fetch`에 페이지 URL 전달
- 페이지 속성 업데이트: `notion-update-page` 사용 (Status 타입 컬럼은 옵션명 그대로 전달)
- 기획 데이터 소스 외 다른 데이터 소스(`collection://3662dc3e-f514-801c-...`, `collection://3662dc3e-f514-809e-...` 등)는 절대 호출하지 않는다.

### Status 타입 컬럼 업데이트 주의사항

상태 컬럼은 `status` 타입이다. 옵션 이름을 정확히 일치시켜 전달해야 한다:
- `기획 전`, `기획 완료`, `검토 필요`, `개발 진행중`, `개발 완료`
- 띄어쓰기 포함 그대로 사용

## 사용자 확인이 필요한 시점

- 최초 프로젝트 설정 시 데이터 소스 URL 확인
- 같은 파일명이 이미 존재할 때 덮어쓰기 여부
- 데이터 소스에 컬럼명/상태명이 일치하지 않을 때 매핑 확인
- 검토 필요 처리 직전 (사용자가 검토 결과를 봐야 하므로 결과만 제시하고 중단)

## 절대 하지 말 것

- 작업 순서 무시하고 건너뛰기
- 검토 필요로 처리하고도 다음 항목 진행
- 기획 외 데이터 소스(TC/QA 등) 읽기/수정
- 기획자가 결정해야 할 사항을 임의로 결정하기 (수치, 규칙, 우선순위 등)
- 노션 페이지를 새로 만들기 (기존 페이지 업데이트만 한다)
- 명세서에 코드 예시/구현 코드 작성 (명세서는 어디까지나 명세이지 구현이 아님)

## 참조 파일

- `references/project_config.md` - 프로젝트별 설정 파일 포맷과 신규 설정 절차
- `references/review_checklist.md` - 검토 필요 판단 기준과 비고 작성 포맷
- `references/dev_spec_template.md` - 개발 명세서 마크다운 템플릿과 작성 원칙
