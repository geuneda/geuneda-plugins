# QA 테스트케이스 Notion DB 생성

## 목적

방금 개발 완료한 기능을, 코드를 모르는 QA 담당자가 그대로 따라 테스트할 수
있는 형태로 만들어 **Notion DB에 행으로 직접 추가**한다. 사용자는 미리 TC
템플릿 페이지를 복제해 둔 빈 페이지의 URL을 제공한다.

## 전제

- 사용자는 본인 Notion 워크스페이스에 **TC 템플릿 페이지**를 먼저 만들어둔다.
  inline database 컬럼 7개 필수:
  - `번호` (title)
  - `항목` (text)
  - `조건` (text)
  - `절차` (text)
  - `기대 결과` (text)
  - `결과` (select: `Pass` / `Fail`)
  - `비고` (text)
- 새 기능마다 그 템플릿을 Duplicate해서 빈 페이지를 만든 뒤, 그 페이지 URL을
  이 스킬에 제공한다.

## 템플릿이 아직 없다면 (첫 사용)

사용자가 워크스페이스에 위 스키마의 템플릿 페이지를 아직 만들지 않았다면,
`AskUserQuestion`으로 두 옵션을 제시한다.

```
[QA TC 템플릿이 없습니다]
어떻게 진행할까요?

1) 에이전트가 템플릿을 복제해 생성한다
   - 복제될 부모 페이지 URL을 입력 받음
   - 아래 공식 템플릿을 notion-duplicate-page로 복제
2) 이미 만들어둔 페이지 URL을 직접 알려준다
```

공식 템플릿 URL (게임팀 - 로켓단 게임즈 / [개발] TC 템플릿):

```
https://www.notion.so/teamsparta/3662dc3ef51480e4aa0afd6780ebcea5
```

이 데이터베이스에는 3개 데이터 소스가 있고, 그 중 **QA 데이터 소스(컬럼:
번호/항목/조건/절차/기대 결과/결과/비고)** 가 이 스킬의 대상이다.

### 옵션 1: 템플릿 복제

1. 사용자에게 복제될 **부모 페이지 URL**을 묻는다.
2. 위 템플릿 URL을 `notion-duplicate-page`로 사용자가 알려준 페이지 아래에 복제.
3. 복제된 페이지의 데이터베이스 자체가 새 TC 페이지로 동작하지는 않는다 —
   `qa-testcase-notion`은 한 기능당 하나의 *복제본 페이지* 단위로 작업하기
   때문이다. 따라서 복제된 페이지의 **데이터베이스 안에서 한 행 단위가
   아니라**, 사용자가 새 기능마다 그 데이터베이스의 빈 사본을 별도로 만들어
   사용해야 한다. 에이전트는 첫 복제 이후 사용자에게 "이번 기능용 빈 페이지
   URL을 알려주세요"라고 다시 묻고 워크플로우 1단계로 진입한다.

### 옵션 2: 사용자가 직접 제공

사용자가 본인 워크스페이스에 이미 빈 TC 페이지를 만들어 두었다면 그 URL을
바로 받아 워크플로우 1단계로 진입한다.

## 워크플로우

### 1. 입력 확인 (사용자에게 직접 확인)

자동 추측하지 않는다. 두 가지를 사용자에게 묻는다.

- **대상 커밋/PR**: 어떤 변경이 TC 대상인지. 예: "최근 자동모드 커밋",
  "PR #1859", "방금 작업한 변경분".
- **Notion 페이지 URL**: 사용자가 TC 템플릿을 복제해 만든 빈 QA 페이지 URL.

이미 사용자가 메시지에 둘 다 줬으면 그대로 진행한다.

### 2. 코드 변경 분석

확정된 범위의 변경을 읽어 **사용자가 체감하는 동작**을 뽑아낸다.

- `git show <ref> --stat`로 변경 파일 파악
- `git show <ref> -- <파일>` 또는 diff로 실제 동작/조건/분기 파악
- 커밋 메시지의 의도 설명을 우선 참고
- 주목할 것: 노출/숨김 조건, 경계값(임계 단계), 자동 진행, 실패 경로,
  설정 저장/복원, 다른 화면에 미치는 파급, 표시(문구/위치)

### 3. 테스트케이스 도출

`references/jargon-glossary.md`를 **반드시** 읽고 그 규칙을 따른다.

- "반드시 포함할 케이스 유형" 8가지를 체크리스트로 빠짐없이 도출
- 개발 용어 → QA 친화 표현 변환 사전 적용 (클래스명/내부 구현 용어 금지)
- 한 케이스 = 한 동작. 사전 조건은 QA가 그 상황을 만드는 방법까지 구체적으로
- 기대 결과는 눈으로 확인 가능한 것만 ("내부적으로", "정상 처리" 금지)

**케이스 정렬 규칙 — 환경 기반 효율 최적화 (3단계)**

QA가 환경(난이도/모드/맵)을 자주 전환하지 않고 한 번 들어간 환경 안에서
묶음으로 검증할 수 있도록 순서를 짠다. "노출→동작→실패" 같은 단순 분류는
환경 전환 비용과 누적 관찰 기회를 무시하므로 쓰지 않는다.

**1단계: 환경 태그 부여**
각 케이스에 검증 가능한 환경 조건을 태그로 단다. 차원은 기능에 따라
조정한다(필요한 것만 쓰고 없는 차원은 생략).

- 진입 환경: 난이도 / 모드(싱글·협동) / 맵
- 기능 내부 상태: 시스템 단계(예: 안전/성장/위험/폭발), 활성/비활성
- 자원 상태: 보유 자원(미네랄/골드/체력) 충분 / 부족
- 게임 진행 단계: 진입 직후 / 진행 중 / 종료
- 선행 조건: 없음 / 이전 케이스의 결과 필요

단순 UI 기능이면 `[노출 조건]` 한 차원만으로 충분. 동적 시스템(이번
화산처럼)은 다 쓴다.

**2단계: 환경 클러스터링 + 전환 비용 최소화 순서**
같은 환경 태그를 공유하는 케이스끼리 묶어 클러스터를 만들고, 환경 전환
비용이 작은 쪽부터 정렬한다.

전환 비용 (작은 것 → 큰 것):
1. 같은 게임 내 시간 진행 (웨이브가 올라가는 것 등)
2. 같은 게임 내 사용자 행동 (버튼 누르기, 자원 소비)
3. 난이도 재시작
4. 협동 모드 진입/이탈
5. 외부 환경 변경 (앱 재시작, 계정 변경 등)

→ 메인 환경(보통 가장 무거운 기능이 동작하는 환경)을 먼저, 부속 환경 검증
(다른 난이도 비노출 같은 것)은 맨 뒤로. 절대 메인 환경 한복판에 끼우지
않는다.

**3단계: 클러스터 내부 정렬 — 누적 동시화 + 의존성 + 실패 경로 배치**

- **누적 관찰 케이스** (시간/웨이브 진행에 따라 자연 변화 — 외형 변화·비용
  변동·자원 누적 등): 같은 시간대에 동시 관찰 가능. 케이스를 통합하지 않되
  **인접 번호로 배치**해 QA가 한 진행에서 같이 체크하도록.
- **선행 조건 있는 케이스** (A의 결과가 B의 사전 조건): 의존 대상 뒤로.
  예: "폭발 후 외형 복귀"는 "폭발 연출" 뒤.
- **실패 경로** (자원 부족, 권한 없음): **환경을 자연스럽게 만들 수 있는
  타이밍**에 배치. 일부러 환경을 만들지 않아도 되는 시점을 노린다.
  예: 게임 초반 미네랄 적을 때 "미네랄 부족 토스트" 검증 → 후반에 와서
  일부러 미네랄 비울 필요 없음.

**결과물: 클러스터 표 + 번호 매김**
도출 직후 클러스터 표(묶음/환경/포함 케이스)를 만든 뒤, 묶음 순서대로
케이스에 번호를 매긴다. 미리보기 단계(4단계)에서 이 클러스터 표도 함께
사용자에게 보여줘 효율을 검증한다.

**항목(item) 작성 규칙**
- **5단어 이내, 짧은 라벨 한 줄로 작성** (예: "화산 외형 확인", "미네랄 부족 토스트")
- 문장형 ❌, "~된다 / ~한다 / ~이 표시된다" 같은 서술 금지
- "조건/절차/기대 결과" 컬럼에 디테일을 담는다. 항목은 인덱스 라벨일 뿐

**비고(note) 작성 규칙**
- **항상 빈 문자열로 둔다**. 비고 컬럼은 QA가 검증 중에 채우는 영역이다.
- 부연 설명이 필요하면 "조건" 또는 "기대 결과"에 녹여 넣는다.

각 케이스를 아래 객체 형태로 메모리에 정리한다(키는 영어, 값은 한국어).

```
{
  "item":         "짧은 라벨 (5단어 이내)",
  "precondition": "사전 조건 (QA가 상황을 만드는 방법)",
  "steps":        "테스트 절차 (1~3문장)",
  "expected":     "기대 결과 (눈으로 확인 가능한 것)",
  "note":         ""
}
```

### 4. 도출 결과 미리보기 → 사용자 승인

Notion에 쓰기 **전에** 두 가지 표를 사용자에게 보여주고 승인을 받는다.

**A. 클러스터 표 (효율 검증용)**
정렬 규칙 3단계의 결과로 나온 묶음을 한눈에 확인할 수 있게 보여준다.
컬럼: `묶음 / 환경 / 포함 케이스 / 동시 관찰·의존성 메모`.

예시:
| 묶음 | 환경 | 포함 케이스 | 비고 |
|---|---|---|---|
| ① 하드 진입 | 하드/안전/초반 | 01 | |
| ② 하드 초반 | 하드/안전/미네랄부족 | 02, 03 | 미네랄 적은 초반에 한 번에 |
| ③ 하드 진행 누적 | 하드/외형 변화 중 | 04, 05 | 동시 관찰 |
| ... | ... | ... | ... |

**B. 케이스 표 (내용 검증용)**
컬럼: `# / 항목 / 조건 / 절차 / 기대 결과 / 비고`.

사용자가 "ok", "응", "ㄱㄱ" 같은 명시적 승인을 줄 때까지 다음 단계로 가지 않는다.
수정 요청이 오면 케이스/클러스터를 고친 뒤 다시 미리보기.

### 5. Notion DB 식별 (data_source_id 추출)

사용자가 준 페이지 URL을 그대로 `notion-fetch`에 넣는다. 결과의 `<content>`
안에서 inline database 태그를 찾는다:

```
<database url="..." inline="true" data-source-url="collection://<UUID>"></database>
```

`collection://` 뒤의 UUID가 **data_source_id**. 이 ID를 `notion-fetch`로
한 번 더 조회해 스키마를 확인한다(컬럼명/타입/select 옵션이 [TC 템플릿]과
동일한지 검증).

기대 스키마:
- `번호` (title)
- `항목` (text)
- `조건` (text)
- `절차` (text)
- `기대 결과` (text)
- `결과` (select: `Pass` / `Fail`)
- `비고` (text)

스키마가 다르면 코드를 진행하지 말고 사용자에게 차이를 보고한다.

### 6. 기존 행 번호 확인

같은 DB에 이미 행이 있을 수 있다. `notion-fetch`로 받은 데이터 소스 상태나
`notion-search`(data_source_url 지정)로 기존 행 개수 N을 파악한다.
새 케이스는 `N+1`부터 번호를 매긴다.

### 7. Notion 행 추가

`notion-create-pages`를 한 번에 호출해 모든 케이스를 행으로 추가한다.

- `parent.type = "data_source_id"`, `parent.data_source_id = <위에서 얻은 UUID>`
- `pages[]`의 각 항목의 `properties`:
  - `번호`: **두자릿수 zero-pad** 문자열 (`"01"`, `"02"`, ..., `"16"`). title 컬럼이 text라 zero-pad가 없으면 "1, 10, 11, ..., 2" 식으로 섞임
  - `항목`: 짧은 라벨 (5단어 이내)
  - `조건`: `precondition`
  - `절차`: `steps`
  - `기대 결과`: `expected`
  - `비고`: **항상 `""`** (QA 영역)
  - `결과`: **설정하지 않음** (QA가 직접 Pass/Fail 채우도록 공란 유지)
- `content`는 주지 않는다(행 본문은 비워둠).

한 번 호출에 100개까지 가능. 100개를 초과할 일은 거의 없지만, 넘으면 나눠서
호출한다.

### 7.5. View 정렬 자동 적용 (번호 ASC)

행 추가 직후, 페이지의 inline DB view에 `SORT BY "번호" ASC`를 자동 적용한다.
이렇게 안 하면 Notion 기본 정렬이 created-time 역순이라 번호가 섞여 보인다.

1. `notion-fetch`로 페이지를 다시 조회해 `<view url="...">` 태그에서 view URL을 추출
2. `notion-update-view`에 `configure: 'SORT BY "번호" ASC'` 전달
3. 실패해도 행 데이터는 이미 들어가 있으므로 사용자에게 "view에서 직접 번호 ASC 정렬 설정" 안내 후 진행

### 8. 결과 보고

생성 완료 후 사용자에게 전달:

- 추가된 행 수 (예: "총 12개 케이스 추가")
- 페이지 링크 (사용자가 준 URL 그대로 다시 보여주기)
- 핵심 테스트 포인트 **3~6줄** 요약 (개발 용어 없이)

## 주의사항
