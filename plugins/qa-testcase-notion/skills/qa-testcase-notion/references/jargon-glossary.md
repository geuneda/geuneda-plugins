# 개발 용어 -> QA 친화 표현 변환 사전

QA 담당자는 코드/구현을 모른다는 전제로 작성한다. 모든 테스트케이스 문장은
"플레이어/사용자가 화면에서 보고 할 수 있는 것"만으로 표현한다.

## 절대 쓰지 않는 단어

코드 구조/내부 구현 용어는 테스트케이스에 절대 등장하면 안 된다.

- 클래스/매니저/서비스 이름 (예: `ArkManager`, `CardSelectionUI`)
- prefab, SerializeField, Inspector, 컴포넌트, 인스턴스
- PlayerPrefs, 캐시, 직렬화, 싱글톤, DI, 바인딩
- async, await, 코루틴, 토큰, 스레드, 콜백, 이벤트 구독
- enum, 상수, 임계값(threshold), 플래그, 파라미터, null
- 메서드/함수명, 변수명, 네임스페이스, 커밋 해시

## 변환 예시 (의미 -> 표현)

| 내부 구현 의미 | QA 테스트케이스 표현 |
|---|---|
| PlayerPrefs에 저장 / 영속화 | "설정이 저장되어 다음에도 유지된다" |
| 앱 재시작 후 값 유지 | "앱을 완전히 종료 후 다시 켜도 그대로다" |
| threshold(임계값) 21 이상일 때 노출 | "21단계 이상을 클리어했을 때만 보인다" |
| CancellationToken으로 취소 | "진행이 멈춘다 / 중단된다" |
| 3초 카운트다운 코루틴 | "3초 후 자동으로 ~된다" |
| 중복 호출 방지 / 디바운스 | "두 번 실행되지 않는다 / 한 번만 처리된다" |
| 상태 강제 해제(reset) | "자동으로 꺼진다 / 초기화된다" |
| 자동 선택 타이머 시작 | "사용자가 누르지 않아도 자동으로 선택된다" |
| Y 오프셋 가산 | "등장 위치(높이)가 다르게 보인다" |
| 보스 제외 분기 | "보스는 기존과 동일하게 동작한다" |
| 로컬라이즈 키로 텍스트 표시 | "문구가 깨지거나 잘리지 않고 보인다" |
| 비활성화/SetActive(false) | "화면에 보이지 않는다" |
| 예외 처리 / 에러 핸들링 | "비정상 종료되지 않고 정상 동작한다" |

## 좋은 테스트케이스의 조건

- 사전 조건: 그 상황을 만들기 위해 QA가 게임에서 해야 하는 준비를 구체적으로
- 테스트 절차: 화면에서 누를 버튼/할 행동만, 1~3문장
- 기대 결과: 눈으로 확인 가능한 결과만. "내부적으로", "정상 처리" 금지 ->
  무엇이 어떻게 보이는지로 환산
- 한 케이스 = 한 가지 동작 검증 (한 행에 여러 시나리오 섞지 않기)

## 반드시 포함할 케이스 유형

코드 변경을 분석할 때 아래 유형을 빠짐없이 도출한다.

1. 정상 동작 (기능이 의도대로 되는 경우)
2. 노출/숨김 조건 (특정 조건에서만 보이는 UI -> 조건 충족 / 미충족 각각)
3. 경계값 (임계 단계의 바로 아래 / 바로 위)
4. 사용자 개입 (자동 진행 도중 직접 버튼을 누르는 경우)
5. 실패/패배/취소 경로
6. 상태 유지 (재진입, 앱 재시작 후 설정 유지)
7. 연관 화면 파급 (이 기능이 켜졌을 때 다른 화면 동작 변화)
8. 표시 품질 (문구 잘림/깨짐, 위치, 애니메이션 정상)
