# TC 내용 구체화 가이드

사용자가 던진 짧은 한 줄을 개발자가 그대로 착수할 수 있는 형태로 다듬는 방법.

## 구체화의 첫 번째 원칙

> 개발자가 이 TC만 보고 재현 + 수정 가능해야 한다.

TC는 "버그가 있어요"가 아니라 "어디서 / 어떤 상태에서 / 무엇을 했을 때 / 무엇이 / 어떻게 잘못됐다"를 담아야 한다. 한 줄짜리 TC는 등록해도 곧 담당자가 다시 물어보러 와야 한다.

## 채워야 할 4가지 항목

모든 TC는 다음 4가지를 갖춰야 한다. 비어 있으면 역질문 후보다.

### 1. 항목 / 대상

영향받는 화면/기능/시스템의 이름. 가능하면 게임 내부 표준 명칭 사용.

- ❌ "버튼이 안 됨"
- ✅ "[항목] 상점 - 일일 패키지 구매 버튼"
- ✅ "[항목] 보스전 BGM"
- ✅ "[항목] 친구 초대 화면 - QR 코드 영역"

### 2. 재현 조건

같은 상황을 만들 수 있도록 필요한 사전 조건. 디바이스 / 계정 / 진입 경로 / 시점.

- ❌ "가끔 그래요"
- ✅ "[재현 조건] iOS 17, 신규 계정, 튜토리얼 완료 직후"
- ✅ "[재현 조건] Android, 챕터 5 보스전 90초 경과 시점"
- ✅ "[재현 조건] 와이파이 → LTE 전환 직후 재화 충전 시"

명시되지 않았지만 일반 상식으로 폭이 좁혀지는 부분은 `(추정)` 표시와 함께 채운다. 너무 광범위하면 역질문.

### 3. 재현 절차 (또는 현재 동작)

번호 매긴 단계. 5단계 이내가 이상적. 개선 TC라면 "현재 동작"을 단계로 적는다.

- ❌ "들어가서 누르면 됨"
- ✅ "[재현 절차]
    1. 메인 화면 → 상점 진입
    2. '일일 패키지' 탭 선택
    3. '구매하기' 버튼 탭
    4. 결제 모달이 떠야 하는데 아무 반응 없음"
- 개선 예시:
  ✅ "[현재 동작]
    1. 친구 목록에서 친구 카드를 누르면 프로필 모달이 뜬다
    2. 모달 닫기 위해 외부 영역을 탭해도 닫히지 않고 X 버튼만 가능"

### 4. 기대 동작 vs 실제 동작 (또는 개선 후 동작)

버그라면 "원래 어떻게 되어야 하는데 실제론 어떻게 되는지". 개선이라면 "어떻게 바뀌면 좋겠는지".

- 버그:
  ✅ "[기대 동작] 결제 모달이 표시되고 결제 진행 가능
     [실제 동작] 버튼은 시각적으로 눌리지만 모달도 안내도 없이 아무 일도 일어나지 않음"
- 개선:
  ✅ "[개선 후 동작] 외부 영역 탭 시 모달이 닫히도록 변경. 다른 모달과 일관성 확보"

## 출력 포맷 (노션 `내용` 컬럼)

`내용` 컬럼은 text 타입이라 마크다운 렌더링이 제한적이다. 줄바꿈은 `\n`을 직접 사용하고 시각적 구분을 위해 항목 헤더에 `[]`를 쓴다.

### 버그 TC 포맷

```
[항목] {기능/화면/시스템 이름}
[재현 조건] {디바이스, 계정, 시점, 기타 환경}
[재현 절차]
  1. {단계}
  2. {단계}
  3. {단계}
[기대 동작] {원래 어떻게 되어야 하는가}
[실제 동작] {지금 어떻게 되는가}
[빈도] {매번 | 가끔 | 1회 관찰}    -- 사용자가 명시했을 때만
[참고] {스크린샷/영상 URL, 관련 로그}  -- 사용자가 제공했을 때만
```

### 개선 TC 포맷

```
[항목] {기능/화면/시스템 이름}
[현재 동작]
  1. {단계}
  2. {단계}
[개선 후 동작] {어떻게 바뀌면 좋은지}
[근거] {왜 개선이 필요한지: 사용자 혼란 / 일관성 / 효율 / 성능 등}
[참고] {레퍼런스, 비슷한 사례}  -- 사용자가 제공했을 때만
```

## 길이 가이드

- 너무 짧으면(`[항목]`만 한 줄, 절차 없음) 거의 항상 역질문 후보
- 너무 길면(10단계 이상 절차, 산문 설명) 단계를 묶거나 항목을 분리해 별도 TC로 나누는 것을 사용자에게 제안
- 일반적으로 5~15줄 사이가 적당

## 추정 표기 규칙

사용자가 명시하지 않았지만 상식적으로 그렇게 보이는 경우 `(추정)` 표시:

- ✅ "[재현 조건] iOS (추정), 신규 계정"
  - 사용자가 "아이폰에서"라고 했는데 OS 버전을 안 적었을 때
- ✅ "[빈도] 가끔 (추정)"
  - 사용자가 "되다 안 되다 그래요"라고만 했을 때

추정이 핵심 항목 전체를 덮으면(예: 재현 절차 전체가 추정) 역질문으로 전환한다.

## 절대 하지 말 것

- 사용자 메시지에 없는 코드/구현 추측 (예: "함수 X가 null을 반환하는 듯")
- 버그 원인 추측 (`원인` 컬럼은 담당자가 처리하며 채움)
- 사용자 표현을 임의로 바꿔서 의미를 비틀기
- 동일한 TC를 여러 항목으로 쪼개기 (사용자가 한 건으로 본 것은 한 건으로 등록)
- 절차에 코드 호출/내부 상태 변경 같은 개발자 행동을 섞기 (TC는 유저 관점 절차여야 함)

## 좋은 예시 vs 나쁜 예시

### 예시 1: 버그

사용자 입력: "보스전에서 궁극기 안 됨"

❌ 나쁜 출력 (그냥 다듬기만):
```
[내용] 보스전에서 캐릭터의 궁극기가 발동되지 않음
```
→ 어떤 보스? 어떤 캐릭터? 매번 그런가? OS는?

✅ 좋은 출력 (역질문 후):
```
[항목] 보스전 - 캐릭터 궁극기
[재현 조건] iOS 17, 챕터 5 보스 (이름: 흑룡), 보스 체력 50% 이하
[재현 절차]
  1. 챕터 5 보스전 진입
  2. 일반 공격으로 보스 체력 50% 이하로 감소
  3. 궁극기 버튼이 활성화되는 것 확인
  4. 궁극기 버튼 탭
[기대 동작] 궁극기 애니메이션 재생 후 대미지 발생
[실제 동작] 버튼은 눌리는 시각 효과가 나지만 애니메이션도 대미지도 없음. 게이지만 0으로 초기화됨
[빈도] 매번 (3회 시도 모두 동일)
```

### 예시 2: 개선

사용자 입력: "친구 초대 화면이 좀 불편함. 코드 복사 버튼 있으면 좋겠음"

✅ 좋은 출력 (큰 역질문 없이 그대로 구체화 가능):
```
[항목] 친구 초대 화면 - 추천인 코드 영역
[현재 동작]
  1. 친구 초대 화면에 추천인 코드가 텍스트로만 표시됨
  2. 유저는 코드를 길게 눌러 텍스트 선택 → 복사 메뉴를 띄워야 함
[개선 후 동작] 코드 옆에 [복사] 버튼을 추가해 한 번 탭으로 클립보드에 복사되고 토스트로 안내
[근거] 길게 누르기 동작이 모든 사용자에게 직관적이지 않고, 다른 게임/앱에서는 복사 버튼 제공이 표준. UX 일관성 + 초대 전환율 개선 기대.
```
