# TC 처리 스킬 (tc-dev-resolver)

`tc-dev-writer`로 노션에 등록된 TC 한 건을 '번호'로 식별해 '내용' 컬럼을 분석하고, 실제 코드베이스를 수정한 뒤 처리 결과를 동일 페이지에 기록하는 후속 단계 스킬이다.

## 핵심 원칙

- **노션이 단일 진실 소스**: TC 내용은 노션 페이지에서 읽는다. 추측이나 사용자 메시지로 보강하지 않는다 — '번호'로 페이지를 찾아 그 행의 '내용' 그대로를 작업 기준으로 삼는다.
- **수정이 불가능하면 솔직하게 '처리 불가'**: 추측으로 패치를 만들거나, 관련 없는 파일을 건드리거나, "일단 한 번 해 봤다"식의 부분 수정을 하지 않는다. 원인을 찾을 수 없거나 코드 외 영역(서버, 아트, 외부 SDK 등)이면 그 사실을 그대로 사용자에게 알리고 노션 상태를 `처리 불가`로 둔다.
- **노션 갱신은 코드 작업이 끝난 뒤에만**: 사용자에게 수정 내용 확인을 받은 후에야 노션 상태/원인/처리 날짜를 갱신한다. 노션 먼저 닫고 코드를 깜빡하는 사고를 막는다.
- **'원인' 컬럼은 한 줄~몇 줄 진단**: 코드 변경 자체가 아니라 "왜 그렇게 됐는지" 한 가지를 적는다. diff는 PR/커밋에 들어간다.
- **TC 외 데이터소스 미수정**: 기획(`collection://3662dc3e-f514-80eb-afe5-000b7e302c1d`) / QA(`collection://3662dc3e-f514-801c-9d9b-000ba701f95b`) 등 TC 외 데이터소스는 절대 읽거나 쓰지 않는다.
- **'선택'/'우선순위'/'내용'은 건드리지 않는다**: 처리자는 등록자가 적은 분류와 본문을 수정하지 않는다. 처리에서 갱신하는 컬럼은 `처리 날짜`, `원인`, `상태` 셋뿐이다.

## 입력 인자

| 위치 | 이름 | 필수 | 설명 |
|------|------|------|------|
| 1 | 번호 | O | 노션 TC의 `번호` (title). 예: `TC-20260520-보스전스킬`. `TC-` 접두는 생략해도 됨. |

인자가 비면 사용자에게 한 번 묻는다. 동일 번호가 여러 건 잡히면 가장 최근(created_time 내림차순) 1건을 보여주고 사용자 확인을 받는다.

## 워크플로우

### Step 1: 프로젝트 설정 확인

`references/project_config.md`에 따라 데이터소스 URL과 컬럼 매핑을 확보한다 (tc-dev-writer와 동일한 설정 파일을 공유한다).

1. 현재 작업 디렉토리에서 `./.tc-dev-writer.json` 확인
2. 없으면 `~/.claude/tc-dev-writer.config.json` 확인
3. 둘 다 없으면 사용자에게 노션 TC 데이터소스 URL을 묻고 새로 작성

설정이 없는 첫 실행에서만 묻는다. 그 다음부터는 자동으로 로드한다.

### Step 2: 노션에서 TC 페이지 조회

`notion-search`로 데이터소스 안에서 `번호`가 입력 인자와 일치하는 페이지를 찾는다 (`references/notion_query.md` 참조).

```
notion-search
  query: "<번호>"
  data_source_url: "collection://<data_source_id>"
  query_type: "internal"
```

매칭 결과 처리:
- **1건**: 그 페이지를 처리 대상으로 확정
- **2건 이상**: 사용자에게 후보 목록을 보여주고 어떤 페이지인지 확인 (URL과 `내용` 첫 80자 미리보기 포함)
- **0건**: 부분 일치 검색으로 한 번 더 시도. 그래도 없으면 "해당 번호의 TC가 노션에 없습니다. 번호 확인 부탁드립니다."로 종료

찾은 페이지에서 다음 컬럼을 모두 읽는다: `번호`, `날짜`, `선택`, `우선순위`, `상태`, `내용`, `원인`, `처리 날짜`.

### Step 3: 이미 처리된 TC 가드

조회 결과의 `상태`가 `완료` 또는 `처리 불가`이면 즉시 사용자에게 안내한다:

```
이 TC는 이미 '{상태}' 상태입니다.

- 번호: {번호}
- 처리 날짜: {처리 날짜 or 없음}
- 원인: {원인 or 없음}

다시 처리하시려면 '재처리'라고 답해주세요. 그대로 두려면 '중단'이라고 답해주세요.
```

`재처리`면 진행, `중단`이면 종료. 사용자 명시 없이 재처리하지 않는다.

`진행 중`이면 그대로 진행한다 (담당자가 이어받는 케이스).

### Step 4: '내용'을 작업 가능한 단위로 해석

`내용` 컬럼은 tc-dev-writer가 생성한 다음 포맷 중 하나다 (`references/content_parsing.md` 참조):

**버그 TC**:
```
[항목] ...
[재현 조건] ...
[재현 절차]
  1. ...
[기대 동작] ...
[실제 동작] ...
```

**개선 TC**:
```
[항목] ...
[현재 동작] ...
[개선 후 동작] ...
[근거] ...
```

각 섹션을 파싱해 다음을 머릿속에 정리한다:

1. **수정 대상 영역**: `[항목]`이 가리키는 화면/시스템/기능 → 코드베이스의 어떤 모듈/파일 후보군인지
2. **재현 시나리오** (버그): `[재현 조건] + [재현 절차]`로 디바이스/계정/진입 경로 식별
3. **차이점** (버그): `[기대 동작] vs [실제 동작]` → 어떤 메서드/상태/조건문이 잘못 됐을 가능성이 있는지
4. **목표 동작** (개선): `[개선 후 동작]`이 요구하는 새 동작과 `[근거]`가 강조하는 일관성/UX 기준

포맷이 위 두 가지에서 크게 벗어나거나 핵심 섹션이 누락됐다면 사용자에게 짧게 확인한다. 노션 페이지를 임의로 수정하지 않는다 — `tc-dev-writer`로 보완 등록을 권한다.

### Step 5: 코드베이스 탐색 및 원인 식별

다음 순서로 진행한다 (`references/resolution_workflow.md` 참조):

1. **위치 추정**: `[항목]`에서 추정되는 키워드(화면명, 기능명, 컴포넌트명)로 `Grep`/`Glob` 탐색
2. **재현 경로 추적**: 사용자가 클릭/탭한 진입점에서 시작해 호출 체인을 따라간다 (UI → 로직 → 데이터 흐름)
3. **차이점 가설**: `[실제 동작]`이 나오는 코드 경로를 찾고, `[기대 동작]`이 되려면 어떤 분기/값/시점이 달라져야 하는지 한 문장으로 정리
4. **원인 확정**: 가설이 코드로 검증되면(해당 라인을 지목할 수 있고, 다른 경로로는 같은 증상이 안 나오면) 원인으로 확정. 검증 안 되면 Step 6 (처리 불가 분기)으로 분기

원인을 정할 때 다음 형태로 한 줄 진단을 만든다:

- "`{파일경로}:{라인}`의 `{함수/조건}`이 `{잘못된 동작}` — `{기대 동작}`이 되려면 `{필요한 변경}`이 필요"
- 예: "`Assets/Scripts/Boss/UltimateSkill.cs:142`의 `OnSkillExecute`가 보스 체력 50% 이하 분기에서 `_animator.Play`를 호출하지 않음 — 게이지 초기화 직전에 애니메이션 트리거가 호출되도록 순서 변경 필요"

추측이거나 "이쪽 어딘가 같다" 수준이면 원인 확정으로 간주하지 않는다.

### Step 6: 처리 불가 분기

다음 중 하나라도 해당하면 코드를 건드리지 않고 '처리 불가'로 마무리한다:

- **코드 외 영역**: 서버/백엔드 API, 외부 SDK 버그, 아트/리소스 파일, OS/디바이스 특이성, 외부 결제 시스템
- **재현 환경 부재**: 명시된 디바이스/계정/시점이 현재 코드에서 재현되지 않고, 재현 없이는 원인 검증 불가
- **TC 자체의 정보 부족**: 코드 탐색을 충분히 했는데도 `[항목]`이 너무 모호해 후보가 너무 많아 좁히지 못함
- **권한/접근 불가**: 노션 권한 또는 코드 저장소 접근이 막혀 작업 불가
- **결정 권한 밖**: 명세를 바꾸는 결정이 필요(기획/디자인 확인 필요)하고 개발자 단독으로 진행할 수 없음

사용자에게 다음 포맷으로 보고하고 노션을 `처리 불가`로 갱신한다:

```
이 TC는 현재 처리 불가입니다.

사유: {한 문장 사유}
탐색 내역: {시도한 파일/검색어/추적한 경로 요약}
필요 조치: {담당자/기획자가 다음으로 할 일 한 줄 — 예: "서버팀 결제 로그 확인", "기획팀 명세 재확인", "디자인팀 리소스 재공유"}

노션 페이지의 '원인'에 이 사유를 기록하고 '상태'를 '처리 불가'로 변경하겠습니다. 진행할까요? (예/아니오)
```

`예`면 Step 8(노션 갱신)으로 넘어가되 상태는 `처리 불가`로 설정한다. `아니오`면 중단.

### Step 7: 코드 수정 및 검증

원인이 확정되면 수정에 들어간다 (`references/resolution_workflow.md` §수정 가이드):

1. **최소 변경 원칙**: 원인이 가리킨 위치를 정확히 수정한다. 주변 리팩토링/스타일 정리는 하지 않는다.
2. **수정 후 자체 검증**:
   - 컴파일/타입 체크가 자동화돼 있다면 돌린다 (예: Unity의 경우 빌드 가능 상태 확인은 IDE/CI에 위임하되 수동 시뮬레이션은 머릿속으로 마친다)
   - 동일 코드 경로의 단위 테스트가 있다면 돌리거나 결과를 사용자에게 알린다
   - 재현 절차를 머릿속에서 한 번 더 시뮬레이션해 새 동작이 `[기대 동작]`과 일치하는지 검토
3. **회귀 위험 검토**: 같은 함수/조건을 호출하는 다른 경로를 한 번 훑고, 부수효과가 새로 생기지 않는지 확인 (없으면 그대로, 의심이 있으면 사용자에게 한 줄로 공유)
4. **수정 요약 작성**: 다음 포맷의 한 단락
   - 변경 파일·라인 / 무엇을 / 왜
   - 예: "`Assets/Scripts/Boss/UltimateSkill.cs:142-148` 분기 순서 변경. 게이지 초기화가 애니메이션 트리거보다 먼저 실행되던 문제를 뒤집어 트리거→초기화 순으로 정리."

수정 직전에 코드 변경 계획을 사용자에게 한 줄로 알린 뒤 진행한다 (Edit 도구 사용). 여러 파일을 손대야 하면 각 파일별로 변경 의도를 함께 공유한다.

### Step 8: 노션 갱신 직전 사용자 확인

코드 변경을 끝낸 뒤 노션에 쓸 내용을 사용자에게 한 번 보여주고 확인을 받는다:

```
다음 내용으로 노션 TC를 갱신하겠습니다:

- 페이지: {URL}
- 번호: {번호}
- 처리 날짜: {YYYY-MM-DD}  (오늘)
- 원인: {Step 5 또는 Step 6에서 정한 한 줄 진단}
- 상태: {완료 | 처리 불가}

코드 변경 요약: {Step 7 요약 한 단락}

진행할까요? (예/아니오/수정)
```

`예`면 Step 9로, `아니오`면 노션을 건드리지 않고 종료, `수정`이면 어떤 부분인지 받아 반영 후 다시 보여준다.

### Step 9: 노션 페이지 갱신

`notion-update-page`로 처리 대상 페이지의 속성 3개를 갱신한다 (`references/project_config.md` §페이지 속성 갱신 호출 예시 참조):

```
notion-update-page
  page_id: <대상 페이지 ID>
  command: "update_properties"
  properties:
    date:처리 날짜:start: "<오늘 YYYY-MM-DD>"
    date:처리 날짜:is_datetime: 0
    원인: "<한 줄 진단>"
    상태: "<완료 or 처리 불가>"
```

`내용`, `선택`, `우선순위`, `날짜`, `번호`는 절대 갱신 대상에 넣지 않는다.

상태 옵션 이름은 정확히 일치해야 한다: `진행 전`, `진행 중`, `완료`, `처리 불가`. 일반 처리 완료는 `완료`, Step 6 분기는 `처리 불가`.

### Step 10: 결과 보고

성공 보고 포맷 (완료):

```
✅ TC 처리 완료

- 페이지: {URL}
- 번호: {번호}
- 원인: {한 줄 진단}
- 처리 날짜: {YYYY-MM-DD}
- 상태: 완료

코드 변경:
- {파일:라인} — {요약}
- (여러 파일이면 각각 한 줄)

다음 단계: 빌드 후 재현 절차로 실제 동작 확인 부탁드립니다.
```

처리 불가 보고 포맷:

```
⚠️ TC 처리 불가

- 페이지: {URL}
- 번호: {번호}
- 원인(사유): {한 문장 사유}
- 처리 날짜: {YYYY-MM-DD}
- 상태: 처리 불가

필요 조치: {다음 담당자/팀에게 넘기는 한 줄}
```

`notion-update-page` 실패 시에는 어디서 실패했는지(권한/컬럼명/옵션명)와 사용자가 다음에 할 일을 안내한다. 코드 변경은 이미 디스크에 있으니 되돌릴지 사용자에게 묻는다 (롤백을 임의로 하지 않는다).

## 노션 MCP 사용 규칙

- TC 페이지 조회: `notion-search` (data_source_url 지정, query_type `internal`)
- 페이지 상세 (속성/본문): `notion-fetch` (페이지 URL)
- 페이지 속성 갱신: `notion-update-page` (command=`update_properties`)
- 페이지 생성/삭제 금지 — 이 스킬은 등록·삭제 권한이 없다 (등록은 `tc-dev-writer`)
- TC 외 데이터소스 호출 금지:
  - 기획: `collection://3662dc3e-f514-80eb-afe5-000b7e302c1d`
  - QA 템플릿: `collection://3662dc3e-f514-801c-9d9b-000ba701f95b`

### 데이터소스 ID 입력 형식

- `notion-search`의 `data_source_url`에는 `collection://` 접두를 포함한다.
  - 예: `collection://3662dc3e-f514-809e-aca8-000bbc3d90ba`
- `notion-update-page`의 `page_id`에는 페이지 자체 URL 또는 UUID를 그대로 사용한다.

### Date 타입 컬럼 expanded format

`처리 날짜` 컬럼은 date 타입이므로 expanded property를 써야 한다:

| 키 | 값 |
|----|-----|
| `date:처리 날짜:start` | ISO-8601 (예: `2026-05-20`) |
| `date:처리 날짜:end` | 단일 날짜면 생략 |
| `date:처리 날짜:is_datetime` | `0` (시각 없음) |

오늘 날짜는 시스템 컨텍스트의 `currentDate`에서 얻는다.

## 사용자 확인이 필요한 시점

- 최초 프로젝트 설정 시 데이터소스 URL 확인 (Step 1)
- 동일 번호 페이지가 여러 건일 때 어떤 것을 처리할지 (Step 2)
- 이미 `완료`/`처리 불가` 상태인 TC를 재처리할지 (Step 3)
- 원인이 확정되지 않아 처리 불가로 마감할지 (Step 6)
- 노션 갱신 직전 최종 확인 (Step 8)

## 절대 하지 말 것

- 노션 페이지의 `내용`, `선택`, `우선순위`, `날짜`, `번호` 갱신 — 처리자는 이 컬럼을 건드리지 않는다.
- TC 외 데이터소스(기획/QA 템플릿) 읽기/수정
- 페이지 자체 삭제 또는 새 페이지 생성
- 사용자 확인 없이 노션 상태를 `완료`/`처리 불가`로 갱신
- 추측만으로 코드 수정 후 `완료` 처리 — 검증되지 않으면 `처리 불가`로 가야 한다
- 원인이 확정되지 않았는데 "일단 의심 가는 곳 한 번 바꿔봄" 식 패치
- 동일 페이지를 같은 세션에서 중복 갱신 (사용자가 명시적으로 다시 처리를 요청한 경우만 허용)
- 처리하지 말라고 했는데도 PR/커밋 생성하기 — 이 스킬은 코드 수정과 노션 갱신까지만 한다 (커밋/푸시는 별도)

## 이 스킬 자체를 업데이트하려면

`references/maintainer.md`를 먼저 읽는다. 핵심 요점:

- 스킬 파일은 사용자의 로컬 작업본(예: `/Users/<id>/geuneda-plugins/plugins/tc-dev/skills/tc-dev-resolver/`)에 있다. 이 스킬은 `tc-dev` 플러그인의 두 스킬 중 하나로, `tc-dev-writer` 스킬과 같은 플러그인을 공유한다. 직접 수정 후 commit + push 하면 팀원이 `/plugin marketplace update geuneda-plugins`로 받아간다.
- 노션 TC 데이터소스 schema가 변경되면 `tc-dev-writer`와 `tc-dev-resolver` 양쪽의 `references/project_config.md`를 동기 갱신한다 (같은 데이터소스를 공유하므로 한쪽만 고치면 어긋난다).
