# 코드 탐색 · 원인 식별 · 수정 · 검증 워크플로우

`내용` 파싱이 끝나 작업 단위가 머릿속에 정리됐다면, 이 문서가 가이드하는 단계로 실제 코드를 손댄다. 목표는 두 가지: **원인을 한 줄로 진단하기**, 그리고 **그 원인만 정확히 수정하기**.

## 코드 탐색

### 위치 추정의 우선순위

1. **`[항목]`의 명시적 키워드**: "보스전 궁극기" → `Boss`, `Ultimate`, `Skill`, 한글 그대로 `보스전`, `궁극기` 등을 `Grep`/`Glob`으로 검색
2. **재현 절차의 진입점**: "X 버튼 탭" → 이벤트 핸들러(`OnClick`, `On...ButtonClick`, `Button.onClick.AddListener`)
3. **데이터 흐름의 종점**: `[실제 동작]`이 나타나는 곳 → 실패 메시지나 잘못된 값을 출력하는 코드 위치
4. **`[참고]`의 스크린샷/로그**: 특정 에러 메시지나 변수명이 보이면 그걸 그대로 검색 (가장 정확한 단서)

순서는 정확도가 보통 큰 순. 1·2가 동시에 잡히면 두 위치의 호출 관계를 따라가며 좁힌다.

### 탐색 도구 선택

- **`Grep`**: 함수명/변수명/문자열 리터럴이 분명할 때
- **`Glob`**: 파일명 패턴(`*Boss*.cs`, `Ultimate*.cs`)으로 좁힐 때
- **`Read`**: 파일이 좁혀졌고 전후 맥락을 봐야 할 때 — 전체 또는 해당 라인 전후 200줄
- **Explore 에이전트**: 후보 파일이 너무 많아 사람이 훑기 어려울 때 (드물게 사용)

탐색 중에는 한 가설을 검증한 뒤 다음 후보로 넘어간다. "이것도 비슷한데" 식으로 여러 후보에 동시에 가설을 세우면 원인이 흐려진다.

## 원인 식별

### 원인 확정 기준

다음 셋을 모두 만족할 때만 "원인 확정"으로 간주한다:

1. **특정 라인을 지목할 수 있다** — `Assets/Scripts/Boss/UltimateSkill.cs:142`처럼.
2. **같은 라인이 `[실제 동작]`을 만든다** — 머릿속에서 재현 절차를 따라가면 그 라인을 거치며 잘못된 결과가 나오는 게 설명된다.
3. **다른 경로로는 같은 증상이 안 나온다** — 같은 함수를 호출하는 다른 분기에서 같은 결과가 나오지 않거나, 나오더라도 같은 라인에서 분기가 갈린다.

세 조건 중 하나라도 모호하면 확정으로 간주하지 않고 **처리 불가** 분기로 넘어간다. "여기일 것 같다"는 추측이 가장 위험하다 — 패치를 만든 뒤에도 진짜 원인이 다른 데 있으면 다음 처리자가 더 큰 비용을 치른다.

### 자주 발생하는 원인 카테고리 (참고)

| 카테고리 | 신호 |
|----------|------|
| 분기 조건 오류 | `>=` vs `>`, null 체크 누락, 잘못된 부울 결합 |
| 호출 순서 잘못 | 상태 초기화가 트리거 전 / 후 잘못된 위치 |
| 상수 잘못 | 0.5 대신 0.05, 챕터 ID 오프바이원 |
| 이벤트 미구독/중복 구독 | UnityEvent/AddListener에서 RemoveListener 누락, 또는 중복 등록 |
| 비동기 타이밍 | async/await 누락, 코루틴 중간 종료, 응답 도착 전 화면 닫힘 |
| 캐시/싱글톤 상태 | 이전 씬 잔여 상태, 정적 변수가 리셋 안 됨 |
| 외부 의존 미주입 | DI 컨테이너 등록 누락, ScriptableObject 참조 끊김 |

카테고리는 머릿속 분류용 — 노션 `원인` 컬럼에는 카테고리 이름이 아니라 구체적 진단 한 줄을 적는다.

## 수정 가이드

### 최소 변경 원칙

- 원인이 가리킨 정확한 위치만 수정한다. 주변 코드의 스타일/네이밍/리팩토링은 하지 않는다 — 다른 PR로 분리.
- 한 라인 수정으로 끝나면 한 라인만. 분기 순서 변경이 필요하면 그 분기만. 새 함수를 빼는 리팩토링은 원래 동작을 보존하는 안전한 경우에만, 그것도 사용자에게 사전 공지.
- 새 종속성/패키지 추가가 필요한 수정이면 사용자에게 먼저 알리고 동의를 받는다.

### Edit 도구 사용

- 단일 라인/블록 수정: `Edit`로 `old_string` → `new_string`
- 한 함수 통째로 다시 작성: `Edit`로 함수 본문을 새 텍스트로 교체
- 새 파일 생성은 거의 발생하지 않는다 — 발생 시 사용자에게 사전 공지

### 자체 검증

수정 직후 머릿속에서 재현 절차를 한 번 다시 돌린다:

1. `[재현 조건]`을 충족한 상태에서
2. `[재현 절차]` 각 단계를 호출 체인으로 따라가며
3. 마지막 단계의 결과가 `[기대 동작]`과 일치하는지 확인

자동화된 검증 수단이 있으면 같이 활용:
- 컴파일/타입 체크: IDE/CI 의존, Claude가 임의로 빌드 명령을 돌리지 않는다 (사용자에게 시간이 걸린다는 안내만)
- 단위 테스트: 같은 코드 경로의 테스트가 있으면 `Bash`로 실행해 결과 공유
- 정적 검사(린트): 프로젝트가 사용하는 도구가 있다면 사용자가 평소 돌리는 명령을 그대로 사용

검증이 한 가지라도 실패하면 수정을 되돌리는 것이 아니라 사용자에게 결과를 그대로 알리고 어떻게 진행할지 묻는다.

### 회귀 위험 검토

수정한 함수/조건을 호출하는 다른 경로를 한 번 훑는다 (`Grep`으로 함수명 검색). 같은 함수가 다른 컨텍스트에서도 호출된다면 변경이 그 컨텍스트를 망가뜨리지 않는지 확인. 의심이 남으면 사용자에게 한 줄로 공유:

```
참고: 같은 함수가 {다른 경로}에서도 호출됩니다.
{변경 의도}는 그쪽에서도 의도와 일치하지만, 빌드 후 그 경로도 한 번 확인 부탁드립니다.
```

## 처리 불가 분기 (재확인)

다음 중 하나라도 해당하면 코드를 건드리지 않는다:

- 코드 외 영역(서버, SDK, 아트, OS) 원인
- 재현 환경 부재 (현재 코드에서 재현 안 됨 + 재현 없이는 원인 검증 불가)
- TC 정보 부족 (탐색 충분히 했는데도 후보를 못 좁힘)
- 권한/접근 불가
- 명세 결정 필요 (개발자 단독으로 진행 불가)

`references/cause_writeback.md`의 처리 불가 사유 작성 가이드를 따른다.

## 수정 요약 작성

사용자에게 보여줄 한 단락:

- 파일 경로 · 라인 번호
- 무엇을 바꿨는지 (한 문장)
- 왜 바꿨는지 (한 문장, 원인의 압축판)
- 다른 영향 (있으면)

예시:

> `Assets/Scripts/Boss/UltimateSkill.cs:142-148`의 분기 순서를 변경했습니다. 보스 체력 50% 이하일 때 게이지 초기화(`_ultimateGauge = 0`)가 애니메이션 트리거(`_animator.Play("Ultimate")`)보다 먼저 호출되어, `Animator.Play`가 이미 0이 된 게이지 값을 참조하는 ScriptableObject의 IsValid 체크에서 false로 떨어지며 애니메이션이 무시되던 문제였습니다. 트리거 → 초기화 순으로 뒤집어 정상화. 다른 호출처(부하 스킬 `MinionUltimate.cs`)는 같은 순서를 이미 따르고 있어 영향 없습니다.

이 한 단락이 `references/cause_writeback.md`의 가이드에 따라 한 줄짜리 `원인`으로 압축된다.
