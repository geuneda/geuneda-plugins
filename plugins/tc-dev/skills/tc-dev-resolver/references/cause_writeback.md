# `원인` 컬럼 작성 가이드

노션 TC의 `원인` 컬럼은 처리자가 진단을 기록하는 자리다. 이 한 줄을 보고 다른 팀원이 "왜 그렇게 됐는지"를 이해할 수 있어야 한다. 코드 변경 자체(diff)는 PR/커밋이 들고 있으므로 여기서는 진단만 적는다.

## 길이와 형태

- **1~3줄**. 한 줄이 기본, 부가 설명이 필요하면 줄바꿈으로 한두 줄 더.
- 평서문. 어떤 함수/조건/순서가 어떻게 잘못됐는지를 단언한다.
- 코드 위치를 가능한 명시. `파일경로:라인` 표기.
- 외부 원인이면 외부 시스템을 명시 (서버팀/SDK/디자인팀 등).

## 좋은 원인 한 줄 예시 (처리 완료)

> `Assets/Scripts/Boss/UltimateSkill.cs:142`의 분기 순서가 잘못돼 게이지 초기화가 애니메이션 트리거보다 먼저 실행되던 문제.

> `Assets/Scripts/Shop/DailyPackage.cs:88`의 구매 버튼 onClick에 PurchaseManager 콜백이 리스너로 등록되지 않아 탭 시 모달이 안 뜸. AddListener 누락.

> `Assets/Scripts/UI/FriendInvite/FriendInvitePanel.cs:55-72`에 복사 버튼 컴포넌트와 ClipboardService 호출을 추가. 기존에는 복사 UI 자체가 없었던 누락 사례.

> `Assets/Scripts/Battle/StageLoader.cs:201`의 챕터 ID 비교가 `>=` 대신 `>`였음. 챕터 5 진입 시 보스 데이터가 한 단계 늦게 로드돼 첫 진입에서 흑룡 대신 이전 챕터 보스 리소스가 사용됨.

### 좋은 이유

- 위치가 분명 (파일:라인)
- 무엇이 잘못됐는지가 분명 (호출 누락 / 순서 잘못 / 비교 연산자)
- "왜 그게 증상으로 이어졌는지"가 한 호흡에 이해됨

## 좋은 원인 한 줄 예시 (처리 불가)

> 결제 모듈은 서버측 응답에 의존하므로 클라이언트 단독 수정 불가. 서버팀 결제 로그 확인 필요.

> 명시된 재현 조건(iOS 17.4, 특정 디바이스)이 현재 빌드/시뮬레이터에서 재현되지 않아 원인 검증 불가. QA가 재현 영상과 디바이스 정보 추가 후 재처리 요청 필요.

> 외부 광고 SDK(AdMob 2.34.0) 콜백 누락 케이스로 추정. SDK 측 변경 사항이라 클라이언트에서 우회 코드는 추가 가능하나 명세 결정 필요. PM 확인 후 재처리.

> `[항목] 보스전 BGM`만 명시돼 어느 보스인지/어떤 상태에서인지 좁혀지지 않음. QA가 보스 이름과 BGM 종류(전투/사망/승리)를 추가해 재등록 필요.

### 좋은 이유 (처리 불가)

- 왜 처리 못 했는지 (외부 / 재현 불가 / 정보 부족 / 결정 권한 밖)가 분명
- 다음 담당자가 뭘 해야 하는지가 한 줄에 같이 들어 있음

## 나쁜 원인 예시 (피해야 함)

### "수정 완료"
**왜 나쁜가**: 진단이 아니라 결과 보고. 다음 사람이 진단을 못 얻음.

### "코드 변경함"
**왜 나쁜가**: 무엇을 어떻게 왜 바꿨는지가 없음. PR이 사라지면 흔적 없음.

### "Unity 버그"
**왜 나쁜가**: 너무 광범위. 어떤 컴포넌트의 어떤 동작인지 없음.

### "버튼 누락 추가"
**왜 나쁜가**: 어디의 어떤 버튼인지가 없음. 같은 화면에 버튼이 여럿이면 다시 찾아야 함.

### "이상해서 일단 수정"
**왜 나쁜가**: 원인 확정이 안 된 상태라는 신호. 이런 진단이 나오면 처리 불가로 가야 함.

### "다음 스프린트에 다시 봐야 함"
**왜 나쁜가**: 진단이 아니라 일정 메모. `원인`이 아닌 별도 코멘트(노션 페이지 본문 등)에 들어가야 함.

## 진단을 한 줄로 압축하는 방법

`references/resolution_workflow.md`의 **수정 요약 작성**에서 한 단락짜리 요약이 만들어진다. 그것을 다음과 같이 압축한다:

1. **위치만 남기고 주변 설명은 제거**: "Unity의 ScriptableObject가 어쩌고..." 같은 배경 설명은 빼고 파일:라인만.
2. **diff 묘사가 아닌 진단 표현으로**: "X를 Y로 바꿨다"가 아니라 "X가 Y라서 Z가 됐다".
3. **원인 한 가지로 좁히기**: 여러 후보가 섞여 있다면 가장 핵심만. 부수적 변경은 PR 본문에.

압축 예:

수정 요약 (단락):
> Assets/Scripts/Boss/UltimateSkill.cs:142-148의 분기 순서를 변경했습니다. 보스 체력 50% 이하일 때 게이지 초기화가 애니메이션 트리거보다 먼저 호출되어, Animator.Play가 이미 0이 된 게이지 값을 참조하는 ScriptableObject의 IsValid 체크에서 false로 떨어지며 애니메이션이 무시되던 문제였습니다.

원인 (한 줄):
> `Assets/Scripts/Boss/UltimateSkill.cs:142`의 분기 순서가 잘못돼 게이지 초기화가 애니메이션 트리거보다 먼저 실행되며 IsValid 체크가 false로 떨어지던 문제.

## 처리 불가 사유 한 줄 작성 패턴

| 카테고리 | 패턴 |
|----------|------|
| 외부 의존 | "{대상 시스템/모듈}이 {이유}로 클라이언트 단독 수정 불가. {다음 담당}에게 {필요 조치}." |
| 재현 불가 | "명시된 재현 조건({조건})이 현재 환경에서 재현되지 않아 원인 검증 불가. {필요한 추가 정보}." |
| 정보 부족 | "{누락된 정보}만으로는 코드 범위 특정 불가. {등록자/QA}가 {추가해야 할 항목} 보완 후 재처리 필요." |
| 결정 권한 밖 | "{변경 사항}은 명세 결정이 필요한 사안으로 개발자 단독 처리 불가. {기획자/PM} 확인 후 재처리." |

## 갱신 시 주의

- 기존 `원인` 값에 내용이 있다면 덮어쓴다. 처리자 간 인계는 별도 PR/슬랙 등 다른 채널을 통하고, 노션 `원인`은 항상 "현 시점의 최종 진단" 한 줄을 유지한다.
- 한국어 위주. 변수명/파일 경로 같은 코드 식별자는 영문 그대로.
- 따옴표/대괄호 같은 텍스트 형식은 그대로 보내도 노션에서 잘 표시된다 — 굳이 이스케이프하지 않는다.
- 길이가 노션 텍스트 컬럼 한도(약 2000자)를 넘을 일은 없다. 넘는다면 진단이 너무 장황한 것 — 더 압축한다.
