# `내용` 컬럼 파싱 가이드

`tc-dev-writer`가 만든 TC의 `내용` 컬럼은 두 가지 포맷 중 하나로 작성된다. 처리자는 이 텍스트를 읽고 머릿속에서 작업 단위로 분해해야 한다. 포맷이 기대와 다르면 임의로 보정하지 않고 사용자에게 확인한다.

## 버그 TC 포맷

```
[항목] {기능/화면/시스템 이름}
[재현 조건] {디바이스, 계정, 시점, 기타 환경}
[재현 절차]
  1. {단계}
  2. {단계}
  3. {단계}
[기대 동작] {원래 어떻게 되어야 하는가}
[실제 동작] {지금 어떻게 되는가}
[빈도] {매번 | 가끔 | 1회 관찰}    -- 있을 때만
[참고] {스크린샷/영상 URL, 관련 로그}  -- 있을 때만
```

### 파싱 → 작업 단위 매핑

| 섹션 | 처리 시 의미 |
|------|--------------|
| `[항목]` | 코드베이스에서 탐색할 모듈/화면/기능의 정체. 첫 검색 키워드. |
| `[재현 조건]` | 분기/플래그/환경 변수 후보. 디바이스/OS/계정 타입에 따라 분기되는 코드가 있다면 그 분기 안에서 본다. |
| `[재현 절차]` | 진입점 → 호출 체인. 단계별 호출되는 함수를 짚어가며 코드 경로를 추적한다. |
| `[기대 동작]` | 수정 후 도달해야 하는 상태. 검증 시 이걸 기준으로 본다. |
| `[실제 동작]` | 현재 코드가 만드는 상태. 어떤 분기/값이 잘못되어 이게 나오는지 가설을 세운다. |
| `[빈도]` | 매번이면 결정적 버그(분기 오류, 상수 잘못), 가끔이면 비결정성(레이스, 외부 응답 의존, 캐시). 가설 우선순위에 반영. |
| `[참고]` | 스크린샷/로그가 있다면 원인 추적을 빠르게 줄여준다. 사용자에게 확인하면 노션에 URL이 들어있을 것. |

### 빈 섹션의 의미

`tc-dev-writer`는 핵심 4섹션(`[항목]`, `[재현 조건]`, `[재현 절차]`, `[기대 동작]`/`[실제 동작]`)을 모두 채운 후에만 등록한다. 그래도 누락되어 있다면 두 가지 가능성이 있다:

1. 등록 후 누군가 수동으로 수정해 비워졌다 → 사용자에게 짧게 확인
2. 등록 시 추정으로 채워진 부분(`(추정)` 표시)이 있다 → 추정 부분이 실제로 맞는지 코드 탐색 중에 검증

## 개선 TC 포맷

```
[항목] {기능/화면/시스템 이름}
[현재 동작]
  1. {단계}
  2. {단계}
[개선 후 동작] {어떻게 바뀌면 좋은지}
[근거] {왜 개선이 필요한지: 사용자 혼란 / 일관성 / 효율 / 성능 등}
[참고] {레퍼런스, 비슷한 사례}  -- 있을 때만
```

### 파싱 → 작업 단위 매핑

| 섹션 | 처리 시 의미 |
|------|--------------|
| `[항목]` | 버그와 동일. 어디를 손댈지의 출발점. |
| `[현재 동작]` | 지금 코드가 어떻게 작동하는지의 사용자 관점 묘사 — 코드의 실제 동작과 일치하는지 먼저 확인 (불일치하면 사실상 버그) |
| `[개선 후 동작]` | 새로 도달해야 하는 동작. 명세에 가까움 — 이게 분명하지 않으면 기획팀 확인 필요(처리 불가 후보). |
| `[근거]` | 변경의 우선순위와 범위를 결정. "다른 화면과 일관성"이면 다른 화면 코드를 참고로 본다. "성능"이면 측정 가능한지 확인. |

### 개선 처리의 추가 주의

- 명세 변경이 큰 경우(새 버튼 추가, 새 화면 진입 흐름) 단독 결정으로 진행하면 안 된다 — 사용자에게 "이건 기획 확인이 필요해 보입니다"라고 한 번 더 확인.
- "예쁘게 / 톤 조정" 같은 시각적 폴리시는 디자인팀 리소스가 따로 필요할 수 있다. 코드 변경만으로 끝나는 범위인지 먼저 판단.

## 포맷 외 케이스

### 둘 다 아닌 자유 형식

가끔 `tc-dev-writer`를 안 거치고 노션에서 직접 작성된 TC가 있을 수 있다. 자유 형식이라도 다음 정보가 추출 가능하면 진행한다:

- 어떤 화면/기능인지
- 어떤 상태일 때
- 무엇이 잘못됐거나 무엇으로 바꾸고 싶은지

위 셋 중 하나라도 빠지면 처리 불가 분기로 넘어가되, "포맷이 표준과 달라 정보가 부족합니다"를 사유로 적는다.

### 본문(page body)에 추가 정보가 있을 때

`tc-dev-writer`는 페이지 본문을 비워두지만, 처리 과정에서 담당자/QA가 본문에 추가 정보(스크린샷, 영상 링크, 로그)를 붙였을 수 있다. `notion-fetch`로 본문도 같이 가져와 단서로 활용한다. 단, 본문은 갱신하지 않는다 (속성만 갱신).

## 작업 단위 추출 결과 예시

다음과 같은 `내용`이 있다고 하자:

```
[항목] 보스전 - 캐릭터 궁극기
[재현 조건] iOS 17, 챕터 5 보스 (이름: 흑룡), 보스 체력 50% 이하
[재현 절차]
  1. 챕터 5 보스전 진입
  2. 일반 공격으로 보스 체력 50% 이하로 감소
  3. 궁극기 버튼이 활성화되는 것 확인
  4. 궁극기 버튼 탭
[기대 동작] 궁극기 애니메이션 재생 후 대미지 발생
[실제 동작] 버튼은 눌리는 시각 효과가 나지만 애니메이션도 대미지도 없음. 게이지만 0으로 초기화됨
[빈도] 매번 (3회 시도 모두 동일)
```

머릿속 작업 단위 정리:

- **탐색 키워드**: "Boss", "Ultimate", "Skill", "Chapter5", 한글로 "보스전", "궁극기"
- **진입점**: 궁극기 버튼 탭 → `OnUltimateButtonClick` 같은 이벤트 핸들러
- **핵심 분기**: 보스 체력 50% 이하 상태 (`bossHp <= maxHp * 0.5`) — 일반 상황과 다르게 동작
- **차이의 후보**: "버튼 시각 효과는 동작 + 게이지 초기화는 동작 + 애니메이션과 대미지만 누락" → 게이지 초기화 호출은 있지만 애니메이션 트리거와 대미지 호출이 안 됨, 또는 호출 순서/조건이 잘못
- **빈도 매번**: 결정적 버그. 분기 조건이나 상수 오류 가능성이 높음 (비결정적 레이스는 후보에서 후순위)
- **검증 기준**: 50% 이하에서 궁극기 버튼 탭 시 (애니메이션 호출 + 대미지 적용 + 게이지 초기화) 셋 모두 실행

이 정리가 끝나면 `references/resolution_workflow.md`의 코드 탐색 단계로 넘어간다.
